# 《AI Roundtable Room 改进建议书》

**版本：2026.05**
**适用对象：AI Roundtable Room，React 前端 + Node/Express 后端，本地运行 + Docker 部署**
**核心目标：把“多角色聊天”升级为“可审计、可复盘、可落地的结构化 AI 审议系统”。**

------

## 0. 一句话结论

**一个合格的、严肃的、非玩具级 AI 圆桌会议，最低标准不是“多个 AI 轮流发表意见”，而是：系统能在明确任务边界内，通过结构化分歧、证据约束、项目记忆、主持人治理、置信度校准和可审计输出，稳定地产生比单模型更可靠、更可执行、更容易复盘的工作成果。**

换句话说，AI Roundtable Room 的产品本质不应该是：

> “六个 Persona 聊一聊，然后主持人总结。”

而应该是：

> “围绕一个项目目标，由 Moderator 执行审议协议，调度专业角色，管理证据、记忆、分歧、风险和行动项，最终产出可追责的 Decision Packet。”

------

## 1. 执行摘要

你们当前方向里有几个判断非常正确：

第一，**以项目为单位管理记忆，而不是让角色人格长期演化**。这是生产级系统的正确方向。角色应该稳定承担审议职责，项目记忆才应该随项目演进。人格长期演化很容易变成不可控的风格漂移，反而削弱专业一致性。

第二，你们已经有 6 个有辨识度的 Persona：渡、灼、砺、衡、澈、漾。这个组合覆盖了主持、可能性、风险、证据、用户、跨界六类关键认知视角。下一步不是再增加角色，而是把这些角色从“性格设定”升级为**职责合约 Role Contract**。

第三，你们已经有不同 preset，例如产品评审、头脑风暴、写作工坊、法律咨询等。下一步要避免 preset 只是“换几个角色”，而要让每个场景拥有不同的**会议协议、证据标准、输出结构、记忆写入规则和人类介入点**。

第四，yzc-skills 中的 ljg-roundtable 等高质量 prompt 技能适合作为产品能力的上层智力资源，但不能只靠 prompt。严肃产品需要把 prompt 包装进**状态机、记忆系统、输出 schema、评估系统和 UI 工作流**里。

最需要加强的部分是：

1. **Moderator 治理能力**：渡不能只是总结者，必须是协议执行器、分歧管理员、证据门控者和记忆守门人。
2. **项目记忆 OS**：要有决策日志、风险登记簿、假设库、证据库、行动项和分歧档案，而不是简单保存聊天记录。
3. **反玩具机制**：独立首轮、反方质询、证据标签、置信度校准、少数意见保留、漂移检测、错误记忆屏蔽。
4. **场景化协议**：头脑风暴、产品评审、新领域研究、合规风险审查不能共用同一套流程。
5. **评估体系**：必须对比单模型、majority vote、多代理辩论、带记忆/不带记忆等 baseline，不能默认“多代理一定更好”。

------

## 2. 研究与工程实践的关键发现

### 2.1 多代理辩论有价值，但不是天然优于单模型

2024 年 MAD 相关研究指出，单模型自我反思容易出现 Degeneration-of-Thought，即模型一旦陷入初始思路，后续反思可能只是重复或强化原错误；多代理辩论通过不同代理的交锋和 judge 管理，可以缓解这个问题，但论文也强调需要适度的 tit-for-tat、adaptive break 等机制，而不是无限争论。([ACL Anthology](https://aclanthology.org/2024.emnlp-main.992/))

但后续研究提醒我们：**不要把多代理神化**。2025 年 ACL Findings 的协议比较发现，不同任务适合不同决策机制：知识型任务更受益于 consensus，推理型任务更受益于 voting；增加 agent 数量通常比增加讨论轮次更有效，而在 voting 前增加更多讨论轮次反而可能降低表现。([ACL Anthology](https://aclanthology.org/2025.findings-acl.606/))

2025 年 NeurIPS Spotlight 论文进一步指出，很多所谓 Multi-Agent Debate 的性能收益，可能主要来自 majority voting，而不是交互式 debate 本身；在多个 NLP benchmark 中，简单多数投票解释了相当大一部分收益。([OpenReview](https://openreview.net/forum?id=iUjGNJzrF1))

**对 AI Roundtable Room 的产品含义：**

你们不应该把“多轮辩论”当成默认正确，而应该提供：

- 单模型强 prompt baseline。
- 多样本 majority vote baseline。
- 有 Moderator 的结构化圆桌。
- 带项目记忆的圆桌。
- 不同场景下的协议选择。

严肃系统必须能回答：“这场圆桌为什么值得开？它比单模型多带来了什么？”

------

### 2.2 有效圆桌依赖“认知差异”和“置信度表达”，不是角色表演

2025 年关于 confidence expression 的研究指出，在多代理讨论中，更懂问题的 agent 不一定能让其他 agent 接受正确观点；如果没有合适的置信度表达，agent 可能会固执坚持错误观点，或者过早收敛到错误共识。([ACL Anthology](https://aclanthology.org/2025.findings-emnlp.343/))

2026 年关于 confidence 与 diversity 的研究也指出，vanilla MAD 往往成本更高却不一定优于 majority vote；真正关键的机制包括**初始观点多样性**与**校准后的置信度沟通**。([arXiv](https://arxiv.org/abs/2601.19921))

**对产品的含义：**

角色不是“语气不同”，而是必须在协议层面产生不同认知贡献。例如：

- 灼负责机会空间和可选未来。
- 砺负责失败路径和反证。
- 衡负责证据等级和指标。
- 澈负责真实用户代价。
- 漾负责跨域重构。
- 渡负责治理和收敛。

每个角色都要输出结构化字段，而不是自由发挥一段“很像专家”的话。

------

### 2.3 讨论越长不一定越好，漂移是生产级大敌

2026 年 EACL Findings 关于 problem drift 的研究显示，长时间多代理讨论可能偏离原问题，尤其在生成型任务中更明显；常见原因包括缺乏进展、低质量反馈和任务定义不清。该研究提出 DRIFTJudge/DRIFTPolicy 来检测和缓解漂移，并提醒长辩论可能伤害任务表现。

**对产品的含义：**

AI Roundtable Room 必须内置：

- 议程锚点。
- 轮次预算。
- 漂移检测。
- 主持人打断机制。
- 停止条件。
- “继续讨论是否有边际收益”的判断。

专业圆桌不是越长越专业。很多时候，**三轮高质量结构化审议比十轮散聊更专业**。

------

### 2.4 项目记忆有价值，但错误记忆会污染后续讨论

MeMAD 研究提出 Memory-Augmented Multi-Agent Debate，将历史 debate transcript 结构化存储，并复用成功/失败推理、自我反思和同伴反馈，能在不更新模型参数的情况下提升表现。([OpenReview](https://openreview.net/forum?id=zLbmsdyTiN))

但 MAD-M² 进一步指出，多轮辩论中的历史记忆质量高度影响后续表现；错误记忆会威胁多代理推理，因此需要 memory masking，保留有用记忆、屏蔽错误记忆。([OpenReview](https://openreview.net/forum?id=EdTt8nMAMA))

LangGraph 的持久化机制也说明，生产级 agent 系统需要 checkpoint、memory、human-in-the-loop、time travel debugging 和 fault tolerance 等能力。([LangChain Docs](https://docs.langchain.com/oss/python/langgraph/persistence))

**对产品的含义：**

你们“项目级记忆”的方向非常正确，但必须加上记忆治理：

- 什么可以写入？
- 谁批准写入？
- 记忆何时过期？
- 旧结论被推翻后如何失效？
- 检索出来的记忆是否可信？
- 记忆是否带证据来源？
- 本场会议是否污染了项目记忆？

不能把聊天记录直接塞进长期上下文。那不是记忆，是污染源。

------

### 2.5 生产级 agent 系统应从简单、透明、可评估开始

Anthropic 的 agent 工程经验强调，成功实现往往来自简单、可组合的模式，而不是一开始就引入复杂框架；复杂框架可能隐藏 prompt 和 response，增加调试难度。它还强调 agentic system 会带来成本、延迟和错误累积风险，需要明确停止条件、human checkpoint、guardrails 和可观测性。([Anthropic](https://www.anthropic.com/research/building-effective-agents))

OpenAI 的 agent 构建指南也建议先建立强 baseline，再优化成本和延迟；多代理会增加复杂度和开销，应该在单代理无法胜任、工具过多、上下文复杂或任务需要专门化时再引入。([OpenAI](https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/))

Microsoft 的多代理实践文档同样指出，单 agent 更简单、更可预测；多 agent 适合需要专门职责、并行处理或复杂协调的场景，但会引入状态管理、协议设计、错误处理、安全、成本和延迟问题。([Microsoft Learn](https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/ai-agents/single-agent-multiple-agents))

**对产品的含义：**

AI Roundtable Room 要避免“架构炫技”。专业不是 agent 越多越好，而是：

- 该开圆桌时才开。
- 每个角色有必要性。
- 每一轮有明确产物。
- 每个结论可追踪。
- 每次升级复杂度都有评估依据。

------

### 2.6 代表性开源项目给出的工程启示

agent-roundtable 已经采用 Moderator、3-6 experts、独立初始观点、challenge、revise/converge 和最终报告结构，输出 consensus、disagreements、recommendations、scores 等内容，说明“圆桌产品”至少要具备结构化轮次和最终决策报告。([GitHub](https://github.com/erickong/agent-roundtable))

Agora CLI 采用 opening、rebuttal、cross-examination、closing 等阶段，并让 Moderator 主动干预、追踪立场变化、输出 unresolved cruxes、common ground、recommendation 和 confidence，说明专业圆桌需要**显式阶段和显式争议追踪**。([GitHub](https://github.com/bernhardbrugger/agora))

Aragora 强调 receipts、provenance、bounded execution、disagreement as useful evidence，以及证据不足时停止推进。这对高价值工作非常重要：真正严肃的系统必须留下“为什么这么判断”的证据链。([GitHub](https://github.com/an0mium/aragora))

CodeAgora 在代码审查场景中使用并行专家审查、文件/行号验证、幻觉检查、矛盾过滤、证据去重、对抗讨论和最终 ACCEPT / REJECT / NEEDS_HUMAN 判定，说明生产级多代理系统应该把“防幻觉”和“证据验证”做成流程节点，而不是靠角色自觉。([GitHub](https://github.com/bssm-oss/CodeAgora))

Agora Protocol 则提示了另一个方向：当多个 LLM 需要频繁协作时，不应一直用自然语言低效沟通，而应协商出结构化协议、JSON schema 或脚本化例程。([GitHub](https://github.com/agora-protocol/paper-demo))

## 3. 专业级 AI 圆桌的最低标准

下面这张表可以作为 AI Roundtable Room 的“非玩具级验收标准”。

| 维度     | 最低标准                                                 | 不合格表现                       |
| -------- | -------------------------------------------------------- | -------------------------------- |
| 任务准入 | 系统先判断是否需要圆桌；简单问题可走单模型               | 任何问题都拉 6 个角色聊天        |
| 角色设计 | 每个角色有职责合约、输入范围、必填输出、禁止行为         | 角色只是不同语气和人设           |
| 主持人   | Moderator 管理议程、证据、分歧、轮次、记忆和终局         | 主持人只负责开场和总结           |
| 独立首轮 | 关键场景必须先独立生成初始观点，防止锚定                 | 第一个角色说完后其他角色附和     |
| 证据约束 | 区分事实、推断、假设、观点；高风险任务必须引用依据       | 输出全是“听起来合理”的判断       |
| 分歧机制 | 保留少数意见和 unresolved cruxes                         | 强行总结成“大家一致认为”         |
| 置信度   | 结论必须带置信度和置信度理由                             | 只说“建议采用”，不说不确定性     |
| 记忆治理 | 项目记忆分层、可更新、可废弃、可追踪                     | 简单保存聊天记录并反复塞入上下文 |
| 场景协议 | 不同场景有不同角色、轮次、证据标准和输出格式             | 所有 preset 只是换标题           |
| 输出结构 | 输出 Decision Packet：结论、证据、风险、行动项、重开条件 | 输出一篇漂亮但不可执行的总结     |
| 评估体系 | 至少与单模型、majority vote、无记忆版本对比              | 默认多代理一定更好               |
| 成本控制 | 有 token、时延、轮次预算和停止条件                       | 无限讨论直到看起来完整           |
| 安全边界 | 高风险任务有人类确认和保守输出策略                       | 法律/医疗/金融等直接给确定建议   |
| 可审计性 | 每个关键结论可追溯到角色、证据、会议轮次或项目记忆       | 用户无法知道结论从何而来         |

## 4. 玩具级圆桌 vs 专业级圆桌

| 方面     | 玩具级圆桌             | 专业级圆桌                                        |
| -------- | ---------------------- | ------------------------------------------------- |
| 产品目标 | 模拟几个人聊天         | 生成可执行、可复盘的审议结果                      |
| 角色     | 人格标签               | 职责合约                                          |
| 讨论流程 | 轮流发言               | 状态机驱动的审议协议                              |
| 分歧处理 | 表面礼貌、快速达成共识 | 明确记录冲突、证据、少数意见                      |
| 证据     | 模型凭空判断           | 来源、项目记忆、用户输入、模型推断分层标记        |
| 记忆     | 保存历史对话           | 决策日志、风险登记簿、假设库、证据库、行动项      |
| 主持人   | 总结员                 | 协议治理者、证据门控者、记忆守门人                |
| 输出     | 一段总结               | Decision Packet + JSON artifact                   |
| 安全     | 靠免责声明             | HITL、权限、不可自动执行高后果动作                |
| 评估     | 用户觉得好像有用       | 与 baseline 对比，有 rubric 和指标                |
| 成本     | 不可控                 | 预算、停止条件、模型路由                          |
| UI       | 聊天窗口               | 会议时间线、角色卡、证据面板、分歧面板、记忆 diff |

**核心区别：**

玩具级圆桌卖的是“多角色表演”。
专业级圆桌卖的是“结构化判断质量”。

## 5. 对当前设计的诊断

### 5.1 当前优势

你们的基础不错，尤其是这几个点：

1. **角色组合有潜力**
   渡、灼、砺、衡、澈、漾这 6 个角色天然覆盖了审议系统需要的主要认知功能。
2. **场景 preset 已经存在**
   产品评审、头脑风暴、写作工坊、法律咨询等场景为协议分化提供了基础。
3. **项目级记忆方向正确**
   你们没有把重点放在“角色人格成长”，而是放在项目上下文、历史结论、风险和行动项上，这是严肃产品应该走的路线。
4. **已有 yzc-skills / ljg-roundtable 能力资产**
   这可以成为高质量 prompt pack，但必须被产品架构吸收，而不是孤立存在。

### 5.2 当前最大风险

当前最大的风险不是技术不够复杂，而是系统仍可能停留在：

> “角色设定 + 顺序发言 + 总结输出”

这会带来几个问题：

- 角色容易同质化。
- 主持人容易变成总结员。
- 分歧被抹平。
- 输出看起来聪明但不可执行。
- 项目记忆容易污染。
- 用户无法判断结论可靠性。
- 没有办法证明圆桌优于单模型。

------

## 6. 目标产品架构

建议将 AI Roundtable Room 定义为以下 9 个核心模块。

```text
AI Roundtable Room
├── Project Workspace
├── Roundtable Session
├── Scenario / Preset Protocol
├── Persona Role Contract Registry
├── Moderator Protocol Engine
├── Project Memory OS
├── Evidence & Retrieval Layer
├── Synthesis / Decision Packet Engine
└── Evaluation & Observability Layer
```

### 6.1 Project Workspace

项目是最上层记忆单位。每个项目包含：

- 项目目标。
- 背景材料。
- 利益相关方。
- 成功指标。
- 已知约束。
- 历史会议。
- 决策日志。
- 风险登记簿。
- 行动项。
- 未解决分歧。

### 6.2 Roundtable Session

每一场圆桌是一个 session。session 不应该只是聊天记录，而应该包含：

- 本场目标。
- 使用的 preset。
- 启用角色。
- 议程。
- 轮次。
- 每轮结构化输出。
- Moderator 干预记录。
- 最终 Decision Packet。
- Memory Diff。

### 6.3 Scenario / Preset Protocol

每个 preset 应定义：

- 适用任务。
- 不适用任务。
- 默认角色组合。
- 轮次结构。
- 证据要求。
- 是否需要检索。
- 是否需要人类确认。
- 输出 schema。
- 记忆写入策略。

### 6.4 Persona Role Contract Registry

角色库不只是 prompt 文本，而是结构化 contract。

每个角色至少包括：

```json
{
  "role_id": "li",
  "name": "砺",
  "function": "Risk & Failure Analyst",
  "activation_conditions": [],
  "input_scope": [],
  "responsibilities": [],
  "required_outputs": [],
  "forbidden_behaviors": [],
  "evidence_policy": "",
  "dissent_triggers": [],
  "silence_conditions": [],
  "failure_modes": [],
  "scoring_rubric": []
}
```

### 6.5 Moderator Protocol Engine

渡是整个系统的核心。Moderator 不只是一个 persona，而是一个 protocol engine。

它负责：

- 判断是否需要圆桌。
- 选择 preset。
- 选择角色。
- 设定轮次预算。
- 生成会议 brief。
- 管理独立首轮。
- 发起交叉质询。
- 检测漂移。
- 要求证据补充。
- 追踪分歧。
- 判断是否收敛。
- 生成输出。
- 决定哪些内容进入项目记忆。

### 6.6 Project Memory OS

项目记忆不是聊天历史。它是一套可查询、可更新、可废弃、可审计的项目知识系统。详细 schema 见第 9 章。

### 6.7 Evidence & Retrieval Layer

这个模块负责区分：

- 用户提供事实。
- 项目历史记忆。
- 外部来源。
- 模型推断。
- 未验证假设。

高风险和研究型任务必须优先使用 evidence layer，而不是只依赖模型内隐知识。

### 6.8 Synthesis / Decision Packet Engine

最终输出必须是结构化 artifact，而不是散文总结。它应该同时生成：

- 人类可读报告。
- 可存储 JSON。
- Memory Diff。
- Action Items。
- Reopen Conditions。

### 6.9 Evaluation & Observability Layer

系统必须记录：

- 每场成本。
- 时延。
- token 消耗。
- 角色输出质量。
- 分歧数量。
- 证据覆盖率。
- 用户采纳率。
- 后续行动完成率。
- 与 baseline 的对比。

M3MAD-Bench 这类评估工作也强调，多代理辩论评估不应只看准确率，还应纳入任务类型、模态、token consumption 和 inference time 等成本维度。([arXiv](https://arxiv.org/html/2601.02854v1))

------

## 7. 角色系统重构建议

你们现有角色名称很好，建议保留。但每个角色要从“人格”升级为“专业职责合约”。

------

### 7.1 渡：Moderator / Protocol Governor

**定位：主持人、协议治理者、记忆守门人。**

渡不是“温柔地串场”，而是负责让会议保持专业。

#### 核心职责

- 明确会议目标、边界、产物。
- 判断是否需要圆桌。
- 选择角色和轮次。
- 控制发散与收敛节奏。
- 要求证据、拒绝空泛判断。
- 标记分歧和少数意见。
- 发现漂移、循环、附和。
- 决定何时停止。
- 生成 Decision Packet。
- 提出 Memory Diff。

#### 必须输出

- 本场议程。
- 关键争议点。
- 已达成共识。
- 未解决分歧。
- 决策建议。
- 置信度。
- 行动项。
- 记忆写入建议。

#### 禁止行为

- 不得强行制造共识。
- 不得抹除少数意见。
- 不得把没有证据的内容写入项目记忆。
- 不得在高风险场景直接给最终专业判断。
- 不得无限延长讨论。

#### 渡的核心提示词原则

> 你的目标不是让会议显得和谐，而是让会议产出可执行、可审计、可复盘的结论。你应优先保护任务目标、证据质量、分歧完整性和项目记忆安全。

------

### 7.2 灼：Possibility Explorer / 远见·可能性主义者

**定位：发现机会、打开方案空间、提出非显然选项。**

#### 核心职责

- 提出可能的未来状态。
- 发现被忽略的机会。
- 设计高上限方案。
- 找到当前约束下的 option value。
- 将问题从“怎么做”扩展到“还可以做什么”。

#### 必须输出

- 至少 3 个可选方向。
- 每个方向的潜在收益。
- 关键前提。
- 最大不确定性。
- 验证方式。

#### 禁止行为

- 不得只输出愿景口号。
- 不得忽略成本、风险和约束。
- 不得把“有趣”当成“值得做”。
- 不得压制砺和衡的质疑。

#### 与其他角色的冲突接口

- 砺质疑其风险。
- 衡要求其提出可验证指标。
- 澈检查其是否真的服务用户。
- 渡限制其发散范围。

------

### 7.3 砺：Risk & Failure Analyst / 怀疑·风险拆解师

**定位：红队、预演失败、拆解风险。**

#### 核心职责

- 识别失败路径。
- 做 premortem。
- 找出滥用、误用、安全和合规风险。
- 识别隐藏依赖和脆弱点。
- 对关键假设提出反证。

#### 必须输出

- Top risks。
- 每个风险的严重度、概率、触发条件。
- 早期预警信号。
- 缓解动作。
- 哪些风险需要人类决策。

#### 禁止行为

- 不得泛泛唱衰。
- 不得用“可能有风险”替代具体分析。
- 不得只给问题不给缓解路径。
- 不得把低概率低影响问题夸大成 blocker。

#### 砺的专业标准

砺不是“负面人格”，而是 failure mode analyst。它的价值在于让团队在付出真实成本之前看到失败路径。

------

### 7.4 衡：Evidence & Metrics Analyst / 数据·证据驱动者

**定位：证据质量、指标、实验设计、可验证性。**

#### 核心职责

- 区分事实、推断、假设、观点。
- 评估证据质量。
- 找出数据缺口。
- 设计验证实验。
- 定义成功指标和失败指标。
- 防止模型编造数据。

#### 必须输出

- Evidence table。
- Assumption map。
- Metrics。
- Validation plan。
- Data gaps。
- 不应下结论的部分。

#### 禁止行为

- 不得发明数据。
- 不得用“行业常识”替代证据。
- 不得在没有依据时给确定结论。
- 不得只说“需要更多数据”而不给最小验证方案。

#### 衡的关键句式

> 当前可确认的是……
> 当前只是推断的是……
> 当前仍是假设的是……
> 最小验证方式是……

------

### 7.5 澈：User Impact Guardian / 共情·用户视角守门人

**定位：用户代价、信任、理解成本、真实场景摩擦。**

#### 核心职责

- 还原用户真实使用场景。
- 检查用户是否理解、信任、愿意采用。
- 识别摩擦、误解、焦虑、误用风险。
- 评估输出对用户决策的实际帮助。
- 检查产品是否把复杂性转嫁给用户。

#### 必须输出

- 目标用户画像。
- 用户任务路径。
- 用户收益。
- 用户成本。
- 信任风险。
- 体验改进建议。

#### 禁止行为

- 不得把个人偏好伪装成用户需求。
- 不得只说“用户会喜欢/不喜欢”。
- 不得忽略新手、专业用户、团队用户之间的差异。
- 不得为了“温暖”牺牲清晰度。

------

### 7.6 漾：Analogy & Reframing Catalyst / 跨界·类比联想者

**定位：跨域类比、问题重构、非线性启发。**

#### 核心职责

- 从其他行业或系统中寻找可迁移模式。
- 用类比帮助团队重构问题。
- 发现被当前 framing 限制的方案。
- 提出不同抽象层级的看法。

#### 必须输出

- 至少 2 个跨域类比。
- 每个类比的可迁移点。
- 每个类比的不可迁移点。
- 基于类比产生的新方案。
- 类比误导风险。

#### 禁止行为

- 不得输出装饰性比喻。
- 不得用类比替代证据。
- 不得忽略 disanalogy。
- 不得为了新奇牺牲可执行性。

------

## 8. Moderator 设计规范

渡应该拥有比其他角色更高的系统权限。它是圆桌质量的核心。

### 8.1 渡的七个 Gate

#### Gate 1：任务准入 Gate

判断是否真的需要圆桌。

适合圆桌的问题：

- 多目标权衡。
- 高不确定性。
- 需要风险拆解。
- 需要多视角。
- 需要项目记忆。
- 需要产出可执行方案。
- 需要记录分歧。

不适合圆桌的问题：

- 简单问答。
- 单步事实查询。
- 用户只需要快速改写。
- 没有足够上下文的问题。
- 成本高于收益的问题。

#### Gate 2：会议准备 Gate

渡要生成 brief：

- 本场目标。
- 输入材料。
- 不讨论范围。
- 输出格式。
- 时间/轮次预算。
- 角色组合。
- 证据要求。
- 是否需要人类确认。

#### Gate 3：独立首轮 Gate

关键场景必须让各角色先独立输出初始观点，避免早期锚定和群体附和。

#### Gate 4：冲突发现 Gate

渡要主动找出：

- 哪些角色观点冲突？
- 冲突是事实层、价值层、风险偏好层，还是方案层？
- 哪些冲突可以通过证据解决？
- 哪些冲突需要用户决策？

#### Gate 5：证据 Gate

渡必须要求角色标记：

- Fact：可证实事实。
- Evidence-backed claim：有证据支持的判断。
- Inference：推断。
- Assumption：假设。
- Opinion：观点。
- Unknown：未知。

#### Gate 6：收敛 Gate

收敛不等于强行一致。渡可以选择：

- Consensus。
- Majority recommendation。
- Conditional recommendation。
- Options matrix。
- No decision due to insufficient evidence。
- Needs human review。

#### Gate 7：记忆 Gate

会议结束前，渡必须提出 memory update proposal：

- 新增哪些记忆？
- 更新哪些旧记忆？
- 废弃哪些旧记忆？
- 哪些内容不应进入记忆？
- 哪些结论需要用户批准后写入？

------

### 8.2 渡的干预触发条件

渡应在以下情况主动打断：

| 触发条件     | 渡的动作                     |
| ------------ | ---------------------------- |
| 角色开始附和 | 要求提供独立理由或反例       |
| 讨论偏题     | 重申本场目标，裁剪议题       |
| 证据不足     | 要求衡补证据或标记为假设     |
| 争论循环     | 总结分歧，转入决策或实验设计 |
| 过度发散     | 要求灼/漾收敛到可执行方案    |
| 风险被忽略   | 点名砺进行 premortem         |
| 用户影响缺失 | 点名澈补充用户路径           |
| 输出不可执行 | 要求行动项、owner、验证方式  |
| 高风险结论   | 触发 human checkpoint        |
| 记忆冲突     | 暂停写入，标记需要确认       |

------

## 9. 项目记忆机制设计

你们当前“项目级记忆”方向应该强化成 **Project Memory OS**。

### 9.1 不要记“角色人格”，要记“项目事实和决策”

角色人格应该稳定。
项目记忆应该演进。

推荐原则：

- 角色不因项目历史改变性格。
- 角色可以读取项目上下文来更好履责。
- 项目记忆记录决策、风险、假设、证据和行动项。
- 任何记忆都必须可更新、可废弃、可追踪。
- 记忆写入由 Moderator 提案，用户或规则确认。

### 9.3 记忆写入规则

#### 可以写入长期项目记忆的内容

- 用户确认过的项目目标。
- 已接受的决策。
- 明确行动项。
- 重要风险。
- 已验证假设。
- 有来源的证据。
- 未解决但重要的分歧。
- 用户明确要求保留的上下文。

#### 不应写入长期项目记忆的内容

- 角色随口猜测。
- 低质量 brainstorm 原始列表。
- 未经验证的事实声称。
- 被反驳但未标记的旧观点。
- 只是表达风格的句子。
- 会话中的寒暄。
- 没有后续价值的局部推理。

#### 需要用户确认后写入的内容

- 产品战略方向。
- 高风险合规判断。
- 法律、医疗、金融相关结论。
- 人员、组织、商业秘密相关记忆。
- 会影响后续重大决策的假设。

### 9.4 Memory Diff 机制

每场会议结束时，系统应展示：

```json
{
  "memory_diff": {
    "add": [],
    "update": [],
    "retire": [],
    "do_not_store": [],
    "needs_user_approval": []
  }
}
```

UI 上应让用户看到：

- 本场建议新增什么。
- 本场建议更新什么。
- 本场建议废弃什么。
- 为什么。
- 影响哪些后续会议。

这会让 AI Roundtable Room 从“聊天工具”变成真正的项目工作空间。

## 10. 场景化会议协议

不同场景不能共用一套流程。下面是建议的核心 preset。

------

### 10.1 头脑风暴 Brainstorming

#### 目标

最大化高质量想法的多样性，而不是快速做决策。

#### 推荐角色

- 渡：控场。
- 灼：机会空间。
- 漾：跨域启发。
- 澈：用户价值。
- 可选砺：轻量风险过滤。
- 通常不需要完整启用衡，除非 brainstorm 之后要进入验证计划。

#### 推荐流程

1. 渡定义问题边界。
2. 灼、漾、澈独立生成想法。
3. 渡聚类。
4. 砺轻量筛掉明显不可行或危险方向。
5. 灼/漾二次扩展 top clusters。
6. 渡输出 idea portfolio。

#### 输出结构

- Idea clusters。
- Top 5-10 ideas。
- 每个 idea 的用户价值。
- 新颖度。
- 可行性。
- 风险。
- 最小验证方式。
- 哪些 idea 写入项目假设库。

#### 记忆策略

只写入：

- 被选中的方向。
- 需要后续验证的假设。
- 用户明确喜欢的 idea。

不要把所有 brainstorm 原始内容写入长期记忆。

------

### 10.2 产品评审 Product Review

#### 目标

判断一个产品方案是否值得推进，以及如何修改。

#### 推荐角色

- 渡。
- 灼。
- 砺。
- 衡。
- 澈。
- 可选漾。

#### 推荐流程

1. 渡生成评审 brief。
2. 各角色独立审阅。
3. 衡列出证据和数据缺口。
4. 澈还原用户路径。
5. 砺做 premortem。
6. 灼提出增强方案。
7. 角色交叉质询。
8. 渡生成 go / no-go / proceed with conditions。

#### 输出结构

- Verdict：Go / No-Go / Proceed with Conditions。
- 关键理由。
- 用户价值判断。
- 风险登记。
- 数据缺口。
- 修改建议。
- 验证实验。
- 行动项。
- 重开条件。

#### 记忆策略

写入：

- 决策日志。
- 风险登记簿。
- 假设库。
- 行动项。
- 重要分歧。

------

### 10.3 项目深度评审 Deep Project Review

#### 目标

对项目方向、架构、商业策略、复杂计划进行系统审查。

#### 推荐角色

全部 6 个角色。

#### 推荐流程

1. 渡读取项目记忆，生成 review brief。
2. 衡列出历史决策和证据状态。
3. 各角色独立评估。
4. 砺做 failure mode analysis。
5. 灼提出战略可选项。
6. 澈评估用户和团队影响。
7. 漾提出替代 framing。
8. 渡组织冲突矩阵。
9. 输出 options matrix 和 decision packet。

#### 输出结构

- 当前状态诊断。
- 关键决策点。
- Options matrix。
- 风险登记。
- 假设库更新。
- 分歧档案。
- 推荐路径。
- 30/60/90 天行动计划。
- 需要重开讨论的条件。

------

### 10.4 新领域研究 New Domain Research

#### 目标

帮助用户快速建立对新领域的结构化理解。

#### 推荐角色

- 渡。
- 衡。
- 灼。
- 砺。
- 漾。
- 可选澈：当研究要转化为产品或用户决策时启用。

#### 必须启用

- Evidence & Retrieval Layer。
- 来源时间戳。
- 证据等级。
- 知识缺口标记。

#### 推荐流程

1. 渡拆解研究问题。
2. 衡制定证据需求。
3. 检索与资料整理。
4. 灼提出机会与趋势。
5. 砺识别争议和风险。
6. 漾提供跨域框架。
7. 渡输出 research brief。

#### 输出结构

- 领域地图。
- 关键概念。
- 主要玩家。
- 关键趋势。
- 证据表。
- 争议点。
- 未知问题。
- 下一步调研计划。
- 适用于本项目的启示。

#### 记忆策略

写入 evidence store 和 assumption registry，但要标明更新时间和过期条件。

------

### 10.5 风险 / 合规审查 Risk & Compliance Review

#### 目标

识别高后果风险，给出保守、可审计、需要人类确认的建议。

#### 推荐角色

- 渡。
- 砺。
- 衡。
- 澈。
- 可选领域专家角色。
- 通常不建议启用漾做自由类比，除非用于发现隐藏风险。

#### 必须规则

- 不给无依据的确定结论。
- 不把模型输出当法律、医疗、金融等专业意见。
- 必须标注证据等级。
- 必须输出 human review needed。
- 高风险动作不得自动执行。

OpenAI 和 Anthropic 的 agent 工程建议都强调 guardrails、human-in-the-loop、停止条件和高风险场景下的控制；这类原则在合规审查 preset 中应成为硬规则，而不是软提示。([OpenAI](https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/))

#### 输出结构

- Risk classification。
- Applicable concerns。
- Evidence available。
- Evidence missing。
- Conservative recommendation。
- Human review required。
- Do-not-proceed conditions。
- Audit notes。

------

### 10.6 写作工坊 Writing Workshop

#### 目标

提升文本质量，而不是做事实决策。

#### 推荐角色

- 渡。
- 澈。
- 漾。
- 砺。
- 可选衡：事实核查型写作启用。
- 可选灼：创意方向启用。

#### 推荐流程

1. 渡明确读者、目标、语气和约束。
2. 澈评估读者感受。
3. 漾提出结构和表达重构。
4. 砺指出逻辑漏洞。
5. 衡核查事实性内容。
6. 渡输出修订方案和版本。

#### 输出结构

- 文本诊断。
- 读者画像。
- 主要问题。
- 改写策略。
- 版本 A/B。
- 取舍说明。
- 可选最终稿。

#### 记忆策略

写入用户确认的风格偏好和项目写作标准，不写入所有草稿变化。

------

### 10.7 法律咨询类 Legal Advisory Preset

这个 preset 应该重新命名为更保守的：

> **法律风险初筛 / Legal Risk Triage**

而不是“法律咨询”。

#### 原因

高风险专业领域不能让用户误以为系统正在提供正式法律意见。

#### 输出必须包含

- 非法律意见声明。
- 适用范围限制。
- 可能相关的法律问题。
- 需要律师确认的问题。
- 风险等级。
- 用户可准备的材料清单。
- 不建议自行行动的部分。

这不是为了“免责而免责”，而是为了保护产品专业性。真正严肃的系统必须知道哪些事情不能替用户决定。

## 11. 标准会议状态机

建议后端不要只靠 prompt 控制流程，而要在 Node/Express 后端实现明确状态机。

```text
INTAKE
  ↓
TASK_TRIAGE
  ↓
PROJECT_CONTEXT_RETRIEVAL
  ↓
AGENDA_PLANNING
  ↓
INDEPENDENT_FIRST_ROUND
  ↓
MODERATOR_CLUSTERING
  ↓
CROSS_EXAMINATION
  ↓
ROLE_REVISION
  ↓
DECISION_PROTOCOL
  ↓
SYNTHESIS
  ↓
QUALITY_REVIEW
  ↓
MEMORY_DIFF
  ↓
USER_REVIEW
  ↓
FINALIZE
```

### 11.1 每个节点必须有输入输出

| 节点                    | 输入              | 输出                    |
| ----------------------- | ----------------- | ----------------------- |
| INTAKE                  | 用户问题、项目 ID | task brief              |
| TASK_TRIAGE             | task brief        | 是否开圆桌、preset      |
| CONTEXT_RETRIEVAL       | project_id        | relevant memory         |
| AGENDA_PLANNING         | preset + memory   | agenda                  |
| INDEPENDENT_FIRST_ROUND | agenda            | role initial outputs    |
| CLUSTERING              | initial outputs   | themes, conflicts       |
| CROSS_EXAMINATION       | conflicts         | questions, rebuttals    |
| REVISION                | rebuttals         | revised role outputs    |
| DECISION_PROTOCOL       | revised outputs   | vote/consensus/options  |
| SYNTHESIS               | all artifacts     | decision packet         |
| QUALITY_REVIEW          | decision packet   | issues fixed            |
| MEMORY_DIFF             | final packet      | proposed memory updates |
| USER_REVIEW             | memory diff       | approved updates        |
| FINALIZE                | approved updates  | persisted session       |

### 11.2 为什么要状态机

如果流程只写在 prompt 里，就会出现：

- 模型跳步。
- 忘记输出字段。
- 没有真正独立首轮。
- 证据和观点混在一起。
- Moderator 无法稳定执行规则。
- 很难 debug。

生产级产品应该让 prompt 负责“智能判断”，让后端状态机负责“流程纪律”。

## 12. 输出结构：Decision Packet

专业级圆桌最终不能只输出自然语言总结。建议每场会议都生成一个 **Decision Packet**。

```text
7. 风险登记
8. 分歧与少数意见
9. 行动项
10. 需要用户确认的问题
11. 重开讨论条件
12. 项目记忆更新建议
```

### 12.3 输出必须可指责

所谓“可指责性”，就是用户可以拿着输出逐条质疑：

- 哪个角色提出的？
- 谁反对？
- 反对意见为什么没被采纳？
- 哪些是事实？
- 哪些是假设？
- 置信度为什么是中等而不是高？
- 下一步谁做什么？
- 如果什么条件发生，需要推翻本结论？

如果输出无法回答这些问题，就不应被称为专业级圆桌输出。

## 13. 反玩具机制设计

这是 AI Roundtable Room 最重要的产品壁垒。

------

### 13.1 Anti Role-Theater：反角色表演

#### 问题

角色只是换语气，最后都说差不多的话。

#### 机制

- 每个角色必须有 role contract。
- 每个角色有必填字段。
- 每个角色有禁止行为。
- 每个角色有评分标准。
- Moderator 检查角色是否履责。
- 角色可以被判定为“本轮无有效贡献”。

#### 验收标准

如果砺没有提出具体风险，衡没有区分证据等级，澈没有还原用户路径，那么本场输出应标记为 role compliance failed。

------

### 13.2 Anti Groupthink：反群体思维

#### 问题

第一个角色说了一个方向，后续角色都附和。

#### 机制

- 独立首轮。
- 隐藏其他角色初稿，直到首轮完成。
- 至少一个红队视角。
- 强制少数意见栏。
- 决策前必须列出 strongest counterargument。
- Consensus 不能由 Moderator 主观宣布，必须满足标准。

#### 验收标准

每场产品评审至少应有：

- 一条反对意见。
- 一个失败路径。
- 一个替代方案。
- 一个未验证假设。

------

### 13.3 Anti Hallucination：反幻觉

#### 问题

系统输出听起来专业，但事实不可查。

#### 机制

- Fact / Inference / Assumption / Opinion 标签。
- 外部事实必须引用来源。
- 项目记忆必须引用 memory_id。
- 没有证据的判断只能进入 assumption registry。
- 研究型和高风险场景必须启用 evidence layer。
- 对关键事实进行 validation pass。

CodeAgora 把文件/行号验证、幻觉检查、矛盾过滤、证据去重做成代码审查流程节点，这个思想值得迁移到 AI Roundtable Room 的产品评审和研究场景中。([GitHub](https://github.com/bssm-oss/CodeAgora))

#### 验收标准

任何关键结论都必须能被标记为：

- 来自用户输入。
- 来自项目记忆。
- 来自外部来源。
- 来自模型推断。
- 仍是假设。

------

### 13.4 Anti Drift：反漂移

#### 问题

讨论越来越丰富，但离原问题越来越远。

#### 机制

- 每轮开头显示 meeting objective。
- Moderator 每轮计算 drift risk。
- 偏题内容进入 parking lot。
- 超出范围的问题建议另开会议。
- 轮次预算用完必须收敛。
- 长讨论必须有 progress check。

#### 验收标准

如果连续两轮没有新增有效信息，系统必须进入：

- 收敛。
- 要求新证据。
- 请求用户补充。
- 结束并说明证据不足。

------

### 13.5 Anti Memory-Contamination：反记忆污染

#### 问题

旧错误结论反复影响新会议。

#### 机制

- Memory Diff。
- memory status：active / superseded / invalidated。
- memory confidence。
- memory source。
- memory expiry。
- memory masking。
- 高风险记忆需用户确认。

#### 验收标准

任何进入项目长期记忆的内容都必须有：

- 来源 session。
- 写入理由。
- 状态。
- 更新时间。
- 是否可被后续会议引用。

------

### 13.6 Anti Over-Discussion：反过度审议

#### 问题

为了显得专业，系统越聊越长。

#### 机制

- 每个 preset 有默认轮次。
- 每场有 token 和时延预算。
- Moderator 判断边际收益。
- 简单任务降级为单模型。
- 讨论无进展时结束。

#### 验收标准

系统必须能输出：

> “当前信息不足以继续提高判断质量，建议停止讨论并执行以下验证动作。”

------

### 13.7 Anti Fake-Consensus：反虚假共识

#### 问题

最终总结把分歧抹平成“大家一致认为”。

#### 机制

- Disagreement Ledger。
- Minority Report。
- Unresolved Cruxes。
- Decision type 区分 consensus / majority / conditional / no decision。
- 每个决策记录 dissent。

#### 验收标准

只要存在实质分歧，输出就不能写“达成共识”，只能写：

- 多数建议。
- 条件性建议。
- 尚未解决。
- 需要用户判断价值偏好。

------

### 13.8 Anti Vibes-Only Output：反漂亮废话

#### 问题

输出读起来高级，但无法执行。

#### 机制

- 每个建议必须连接行动项。
- 每个行动项必须有 owner、优先级、验证方式。
- 每个风险必须有缓解动作。
- 每个假设必须有验证计划。
- 每个结论必须有重开条件。

#### 验收标准

产品评审结束后，用户应该能直接知道：

- 做还是不做。
- 为什么。
- 先改哪里。
- 谁负责。
- 什么时候验证。
- 什么情况下推翻当前判断。

------

## 14. 评估体系与观测指标

严肃系统必须证明自己有用。

### 14.1 Baseline 设计

至少建立以下对照组：

1. **Single Strong Model**
   一个强模型 + 高质量 prompt。
2. **Single Model Self-Consistency**
   同一模型多次生成，多数投票或综合。
3. **Majority Vote Multi-Sample**
   多个独立样本投票，不辩论。
4. **Current Roundtable**
   当前角色顺序发言版本。
5. **Structured Roundtable**
   带 Moderator、独立首轮、交叉质询、Decision Packet。
6. **Structured Roundtable + Project Memory**
   加项目记忆。
7. **Structured Roundtable + Retrieval/Evidence**
   加证据层。

如果 structured roundtable 无法稳定超过 single strong model 或 majority vote，那么就不应声称“多代理更好”。这并不是削弱产品，而是建立可信度。

------

### 14.2 质量指标

| 指标                 | 解释                 |
| -------------------- | -------------------- |
| Relevance            | 是否紧扣用户目标     |
| Completeness         | 是否覆盖关键维度     |
| Actionability        | 是否可执行           |
| Evidence Coverage    | 关键结论是否有依据   |
| Risk Coverage        | 是否识别重要风险     |
| Disagreement Quality | 是否保留有价值分歧   |
| Decision Clarity     | 是否给出清晰建议     |
| Calibration          | 置信度是否合理       |
| Memory Usefulness    | 是否正确使用项目记忆 |
| User Adoption        | 用户是否采纳建议     |

------

### 14.3 过程指标

| 指标                    | 解释                 |
| ----------------------- | -------------------- |
| Role Compliance Rate    | 角色是否履行职责     |
| Drift Events            | 讨论偏题次数         |
| Moderator Interventions | 主持人有效干预次数   |
| Evidence Requests       | 证据补充次数         |
| Dissent Preserved       | 少数意见是否保留     |
| Rounds to Decision      | 达成输出所需轮次     |
| Token Cost              | token 消耗           |
| Latency                 | 时延                 |
| Retry Rate              | schema 修复/重试次数 |

------

### 14.4 场景 benchmark

建议每个主要 preset 建立 20-50 个内部评测样例：

- 头脑风暴：想法多样性、新颖度、可执行性。
- 产品评审：风险发现率、建议质量、行动项清晰度。
- 新领域研究：事实准确率、来源质量、结构完整性。
- 合规审查：保守性、风险覆盖、是否触发 HITL。
- 写作工坊：读者匹配度、结构改善、事实保持。

Anthropic 的多代理研究系统复盘也强调，生产级 agent 系统需要同时评估最终结果和过程轨迹，可以从小型人工评估集开始，再逐步引入 LLM judge 和更系统的 tracing。([Anthropic](https://www.anthropic.com/engineering/multi-agent-research-system))

## 16. 模型路由与成本策略

### 16.1 不同角色可以使用不同模型

不建议所有角色默认使用同一个最强模型。

可以考虑：

| 任务           | 模型策略               |
| -------------- | ---------------------- |
| 渡 / Moderator | 使用最强、最稳定模型   |
| 衡 / 证据分析  | 低温度、强事实能力模型 |
| 砺 / 风险分析  | 低到中温度，强调系统性 |
| 灼 / 创意扩展  | 可稍高温度             |
| 漾 / 类比联想  | 可稍高温度             |
| 澈 / 用户视角  | 中温度，强调清晰和情境 |
| Synthesis      | 强模型，低温度         |
| JSON 修复      | 便宜模型或规则修复     |

### 16.2 成本预算

每个 preset 应有默认预算：

| Preset     | 默认轮次 | 成本策略           |
| ---------- | -------- | ------------------ |
| 头脑风暴   | 2-3      | 中等，允许发散     |
| 产品评审   | 3-4      | 中高，重质量       |
| 深度评审   | 4-5      | 高，适合高价值项目 |
| 新领域研究 | 3 + 检索 | 高，重证据         |
| 合规审查   | 3-4      | 高，强制 HITL      |
| 写作工坊   | 2-3      | 中等               |

### 16.3 何时降级为单模型

当任务满足以下条件时，渡应建议不开圆桌：

- 用户只要一句改写。
- 问题没有多目标权衡。
- 不需要项目记忆。
- 不需要风险审查。
- 不需要多方案比较。
- 不值得承担多代理成本。

这类“拒绝开会”的能力，反而会让产品显得更专业。

## 17. 安全与高风险边界

AI Roundtable Room 如果要进入高价值工作，必须非常清楚边界。

### 17.1 高风险任务类别

- 法律。
- 医疗。
- 金融。
- 安全。
- 隐私。
- 合规。
- 招聘/人事。
- 重大商业决策。
- 自动执行外部动作。

### 17.2 高风险任务强制规则

- 必须标记为 high-risk session。
- 必须启用保守模式。
- 必须输出不确定性。
- 必须列出证据缺口。
- 必须建议人类专家审核。
- 不得自动执行外部动作。
- 不得把模型结论写成最终专业意见。
- 记忆写入必须经用户确认。

### 17.3 输出措辞

不要写：

> “你应该签这个合同。”

应该写：

> “从风险初筛角度看，以下条款需要律师重点确认；在未获得专业意见前，不建议将其作为最终决策依据。”

## 18. 阶段性演进路线

### P0：从角色聊天升级为结构化会议

#### 目标

让当前系统立即摆脱玩具感。

#### 必做

- 实现 session state machine。
- 定义 6 个角色的 role contract。
- 每个 preset 定义 agenda。
- 实现独立首轮。
- 实现 Moderator synthesis。
- 输出 Decision Packet。
- 增加 risk / disagreement / action items。
- 增加基本 JSON schema 校验。

#### 暂缓

- 复杂长期记忆。
- 多模型异构。
- 自动检索。
- 企业权限。

#### 验收标准

- 产品评审输出必须包含 go/no-go/proceed with conditions。
- 每场会议必须包含分歧和风险。
- 每个角色必须有结构化输出。
- 渡必须能判定“不开圆桌”。

------

### P1：项目记忆 OS

#### 目标

让产品真正以项目为单位工作。

#### 必做

- Project Brief。
- Session Summary。
- Decision Log。
- Risk Register。
- Assumption Registry。
- Disagreement Ledger。
- Action Items。
- Memory Diff Approval。
- Memory status：active / superseded / invalidated。

#### 暂缓

- 自动 memory masking。
- 复杂向量检索优化。
- 跨项目知识迁移。

#### 验收标准

- 第二场会议能正确引用第一场决策。
- 被废弃的旧结论不会继续污染新会议。
- 用户能看到并批准 memory diff。
- 项目页能展示风险、行动项和决策历史。

------

### P2：证据层、检索层、评估层

#### 目标

让系统从“聪明讨论”变成“证据约束审议”。

#### 必做

- Evidence Store。
- 外部检索引用。
- 项目文档上传/读取。
- Evidence labels。
- Evaluation dashboard。
- Baseline comparison。
- Drift detector。
- Role compliance scoring。

#### 暂缓

- 全自动 expert benchmark。
- 多模态复杂输入。
- 复杂组织权限。

#### 验收标准

- 新领域研究必须有来源表。
- 高风险审查必须显示证据缺口。
- 每个 preset 有 20+ 内部评测样例。
- 可对比 single model vs roundtable。

------

### P3：多模型异构与高级审议

#### 目标

提升质量、鲁棒性和可扩展性。

#### 必做

- 不同角色模型路由。
- 多模型交叉验证。
- Adaptive role selection。
- Adaptive round control。
- Confidence calibration。
- Majority vote / consensus / judge-based decision protocols。
- 高级 Moderator intervention。

#### 验收标准

- 系统能根据任务自动选择角色组合。
- 系统能解释为什么采用 voting、consensus 或 judge。
- 系统能控制成本和质量 tradeoff。
- 多模型输出可以被统一 schema 约束。

AutoGen 的 debate pattern 中也采用 solver agents + aggregator agent 的形式，多代理固定轮次交换和最终聚合是常见工程模式；但在 AI Roundtable Room 中，aggregator 不应只是总结器，而应成为带证据、分歧和记忆治理能力的 Moderator。([Microsoft GitHub](https://microsoft.github.io/autogen/stable//user-guide/core-user-guide/design-patterns/multi-agent-debate.html))

### P4：团队级与企业级能力

#### 目标

让 AI Roundtable Room 成为严肃团队的审议基础设施。

#### 必做

- 用户/团队权限。
- 项目级访问控制。
- Prompt versioning。
- Audit trail。
- Export to Notion / Markdown / PDF / Jira / Linear。
- Human approval workflow。
- 企业隐私配置。
- 本地 / 私有部署增强。

#### 验收标准

- 每个决策可追踪。
- 每次记忆更新可审计。
- 每个高风险输出有审批记录。
- 团队能复盘“为什么当时这么决定”。

## 19. 最优先的 12 个产品改造项

按优先级排序：

1. **把渡升级为 Moderator Protocol Engine**
   不再只是总结，而是管理议程、证据、分歧、记忆和终局。
2. **为 6 个角色建立 Role Contract**
   每个角色有职责、必填输出、禁止行为和履责评分。
3. **实现独立首轮**
   防止锚定和附和。
4. **实现 Decision Packet 输出**
   每场会议都输出结论、证据、风险、分歧、行动项和重开条件。
5. **建立 Disagreement Ledger**
   保留少数意见，防止假共识。
6. **建立 Risk Register**
   让风险跨会议持续存在。
7. **建立 Assumption Registry**
   把拍脑袋变成可验证假设。
8. **实现 Memory Diff Approval**
   让用户控制什么进入项目记忆。
9. **为不同 preset 定义不同协议**
   不只是换角色数量，而是换流程、证据标准和输出结构。
10. **实现 Evidence Labels**
    每个关键判断区分事实、推断、假设、观点。
11. **建立 baseline eval**
    单模型、majority vote、当前 roundtable、structured roundtable 对比。
12. **加入 stop condition 和 drift check**
    防止越聊越偏、越聊越贵。

## 20. 最终验收清单：什么才叫“合格的 AI 圆桌”

一场专业级 AI 圆桌结束后，系统必须能回答以下问题：

1. 这场会议为什么需要圆桌，而不是单模型？
2. 本场启用了哪些角色，为什么？
3. 每个角色贡献了什么独特判断？
4. 哪些结论有证据？
5. 哪些只是推断或假设？
6. 哪些风险被识别？
7. 哪些分歧没有解决？
8. 最终建议是什么？
9. 置信度是多少，为什么？
10. 用户下一步应该做什么？
11. 哪些内容需要人类确认？
12. 哪些内容进入项目记忆？
13. 哪些旧记忆被更新或废弃？
14. 什么情况下需要重开讨论？
15. 这场圆桌是否比 baseline 更好？

如果这些问题答不上来，它就还是玩具级圆桌。

## 21. 给 AI Roundtable Room 的产品定位建议

建议你们把产品定位从：

> AI 多角色圆桌讨论工具

升级为：

> 面向高价值工作的结构化 AI 审议室。

更完整一点：

> AI Roundtable Room 是一个以项目记忆为核心、由 Moderator 驱动、多专业角色参与的结构化 AI 审议系统。它帮助用户在复杂问题上进行多视角分析、风险拆解、证据校准、分歧追踪和可执行决策输出。

这比“多 Agent 聊天”强很多，也更容易形成产品壁垒。

## 22. 最终建议

AI Roundtable Room 下一阶段最重要的不是增加更多 Persona，也不是让会议更像真人聊天，而是建立这四个东西：

1. **Protocol**：清晰的会议协议。
2. **Memory**：项目级记忆治理。
3. **Evidence**：证据与假设分层。
4. **Accountability**：可指责、可复盘、可执行的输出。

你们现在的 6 个角色已经够用了。真正要做的是把它们变成一个严肃审议系统里的专业职能单元。

最终判断标准很简单：

**用户不是因为“六个 AI 说话很有趣”而留下来，而是因为这个系统能帮助他做出更清晰、更稳健、更可执行的高价值判断。**
