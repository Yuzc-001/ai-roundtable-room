# AI Roundtable Room UI 大改设计方案（v1）

**日期**：2026-05-17  
**目标**：将当前“功能堆砌、视觉混乱”的界面，彻底重构为一个**专业级、项目级结构化 AI 审议工作空间**。

**核心定位**（来自调研01 + 调研02 + 产品目标）：
AI Roundtable Room 不是“多个 AI 角色聊天”，而是**为具体项目提供结构化审议、证据管理、分歧保留、记忆累积和可审计决策**的工具。

我们要做的是 Deliberation（审议），而不是 Debate（辩论）或简单聊天。

---

## 1. 整体信息架构（三层空间模型）

### 1.1 三层结构

1. **Project Workspace（项目工作空间）** —— 最外层，常驻
   - 项目列表与切换
   - 项目概览（目标、约束、成功指标、背景材料）
   - 项目记忆总览（决策日志、风险登记簿、行动项、未解分歧、假设库）
   - 项目文件/资料区
   - 历史圆桌列表（可快速进入或复盘）

2. **Roundtable Session（圆桌会场）** —— 进入具体一场会议后的状态
   - 采用明确的“进入会场 / 退出会场”机制
   - 支持 **Focus Mode（专注模式）**：一键隐藏项目列表、记忆面板等，只保留核心会场
   - 阶段化呈现（见下方）

3. **Review & Output（审核与产出区）** —— 独立或半独立工作流
   - Memory Review（记忆变更审核，支持 Propose-Promote）
   - Decision Packet 最终确认、导出、回写
   - 记忆 Diff 可视化

**设计原则**：不同层级的信息不要同时高强度竞争注意力。进入会场后，项目管理要退居二线。

---

## 2. 整体布局方向（桌面端）

**推荐布局（三栏 + 聚焦）**：

- **左侧固定/可收起栏（280-320px）**：Project Workspace
  - 项目列表
  - 当前项目关键信息快照
  - 记忆摘要（Top 风险 + 待办行动 + 未解分歧数量）

- **中间主区域（占据 55-60% 宽度）**：Roundtable Session（会场核心）
  - 顶部状态栏：当前议题 + 当前场景（Preset） + 阶段指示器 + 主持人控制入口
  - 角色席位区（重新设计，克制正式）
  - 讨论时间线（支持结构化展示：act 类型、证据标记、Tension 关联）
  - 底部/浮动：Moderator 主动控制区

- **右侧可折叠栏（240-280px）**：当前会场上下文
  - 本场 Tensions（分歧）列表（带状态）
  - Evidence Pool（本场证据）
  - 项目记忆快速引用
  - 当前阶段提示

**Focus Mode（强烈推荐实现）**：
- 一键进入：隐藏左侧项目栏 + 右侧大部分面板，只保留角色席位 + 时间线 + 最小 Moderator 控制
- 适合长时间深入审议

## 3. 场景（Preset）差异化设计

不同场景不能只是“角色数量不同”，必须在**视觉、节奏、交互、协议**上都有明显差异。

### 3.1 四个核心场景建议

**1. 头脑风暴（Brainstorm）**
- 视觉：轻盈、留白多、柔和色彩
- 节奏：快速发散 → 快速聚类
- 交互重点：快速输入、拖拽聚类、点赞/弱反对
- 角色：灼（远见）+ 漾（跨界）+ 澈（用户）+ 渡（控场，抑制过早批判）
- 阶段侧重：Diverge 占比极高，Examine 弱化
- 反玩具：限制过早进入收敛

**2. 项目深度评审（Project Review）**
- 视觉：正式、结构清晰、适度留白
- 节奏：均衡，强调证据与冲突
- 交互重点：证据标记、Tension 高亮、投票/立场表达
- 角色：全员（灼+砺+衡+澈+漾 + 渡）
- 阶段侧重：Surface + Examine + Converge
- 反玩具：强制每轮异议、证据要求

**3. 新领域研究（New Domain Research）**
- 视觉：学术感、知识图谱倾向
- 节奏：多次 Frame + Surface 迭代
- 交互重点：知识缺口可视化、假设追踪、主动 PROBE
- 角色：衡（证据）+ 灼（可能性）+ 漾（类比）+ 渡 + 可选外部专家
- 阶段侧重：Surface + Examine 强化

**4. 风险/合规审查（Risk & Compliance）**
- 视觉：严肃、警示色（红/橙强调）
- 节奏：慢、穷举、红队
- 交互重点：强制 Devil’s Advocate 每轮、风险严重度标记、合规 checklist
- 角色：砺（核心）+ 衡 + 渡 + 临时注入“对手方”（红队）
- 阶段侧重：Examine 极度强化
- 反玩具：每轮必须有 OBJECTION

**实现方式**：
- 每个 Preset 对应一套 Protocol 配置（角色组合、阶段权重、证据要求、匿名化开关、Moderator 行为偏好）
- 视觉系统根据 Preset 切换颜色基调、卡片样式、留白程度

## 4. 角色呈现方式重构（C）

当前 Stage 的 3D 倾斜设计在专业产品中容易显得花哨且信息密度低，建议大幅简化。

### 4.1 新角色呈现原则

- **克制、正式、可读性优先**
- 采用半圆或水平列表式席位（可切换）
- 每个角色卡片必须展示：
  - 名称 + 标题 + 当前状态（发言中 / 活跃 / 静默 / 已贡献关键证据）
  - 本场累计贡献（已提 CLAIM 数、OBJECTION 数、EVIDENCE 数）
  - 当前立场摘要（可折叠）
- 支持点击角色快速查看其“本场贡献时间线”

### 4.2 角色状态可视化

- 颜色 + 图标区分状态
- 特别高亮“提出过重要 OBJECTION”或“贡献了关键证据”的角色
- 静默角色要明显（这是健康信号，不是 bug）

## 5. Moderator（渡）操作界面与权限（D）

这是目前 UI 最弱的一环，必须大幅加强。

### 5.1 独立 Moderator 控制台

- 给渡一个**常驻或易于呼出的强大控制面板**（类似会议主持人控制台）
- 核心功能：
  - 当前阶段 + 阶段推进 / 回退按钮
  - 主动发起 PROBE（信息缺口探测）
  - Tension（分歧）列表管理（查看、要求回应、标记解决）
  - 漂移警告与干预
  - 手动要求特定角色回应
  - 生成阶段总结
  - 记忆写入提案（Propose）
  - 人类闸门触发

### 5.2 人类闸门设计

- 高风险操作（进入 Decide、写入高影响记忆、合规相关结论）必须有明显的人类确认步骤
- UI 上用颜色和尺寸区分“AI 可自动”与“必须人类确认”

---

## 6. 其他必须一起解决的 E 部分

- **Focus Mode（专注模式）**：强烈建议作为核心功能。一键进入后大幅减少视觉噪音。
- **Tension 可视化**：分歧不能只藏在聊天里，要有独立面板 + 关系视图。
- **Evidence 面板**：独立区域展示本场 + 项目级证据，支持快速引用。
- **Memory Diff 可视化**：要做得清晰好看，让用户愿意认真审核（这是项目记忆价值的关键）。
- **不同场景的节奏差异**：
  - 头脑风暴：支持快速输入和拖拽聚类
  - 项目评审：强调证据链和冲突追踪
  - 风险审查：每轮强制红队视觉提示
- **可扩展组件系统**：建立清晰的原子组件（RoleCard、TensionItem、EvidenceTag、DecisionPacketPreview 等），避免继续堆屎山。

## 7. 组件拆分建议（工程层面）

建议把当前巨大 App.jsx 拆分为：

- ProjectWorkspace
- RoundtableSession（核心容器）
- RolePresentationLayer
- ModeratorControlPanel
- TensionBoard
- EvidencePanel
- MemoryReviewPanel
- DecisionPacketRenderer
- FocusModeManager

配合清晰的状态机管理会话阶段（Frame / Diverge / Surface / Examine / Converge / Decide / Document）。
