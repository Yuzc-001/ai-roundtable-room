# AI Roundtable Room 改进建议书

> 一份关于"严肃 AI 圆桌"最低标准与可指责性架构的设计文档
>
> 调研时间窗口：2024.01 – 2026.05 适用对象：AI Roundtable Room 产品架构、提示词工程与项目记忆模块

------

## 00 / 立场声明

### 0.1 我们到底在做什么

AI Roundtable Room 不是"多个 AI 角色轮流聊天"。这种东西在 2025 年的 GitHub 上已经有几十个原型，绝大多数都活不过三个 demo。如果产品的内核仍然是"轮流发言 + 最终总结"，那么无论 UI 多精致、角色名字多好听，它都是玩具。

真正值得做的，是 **deliberation tool（审议工具）**。这是一个 2026 年才在学术上被正式形式化的范畴（Prakash, *From Debate to Deliberation*, arXiv:2603.11781），与 debate / orchestration / ensembling / voting 都不同：

- **Debate** 关心谁赢；
- **Orchestration** 关心任务分发；
- **Voting** 关心多数结论；
- **Deliberation** 关心"在差异化参与者之间，按类型化的认知动作交换信息，把分歧保留为生产性张力，在受控规则下接纳新证据，最终收敛为可问责的决定"。

AI Roundtable Room 的产品定位必须在第四种里。这是它存在的唯一合理理由——因为前三种已经各有更好的工具占领。

### 0.2 玩具 vs 专业的可证伪判据

下面这张表是整份建议书的"地基"。后续每一支柱都对应表中的一行，并给出可被检查的最低标准。任何一行停在左列，整个产品就是玩具。

| 维度         | 玩具级                               | 专业级最低标准                                               |
| ------------ | ------------------------------------ | ------------------------------------------------------------ |
| 1 交互范式   | 角色轮流自由发言，最后 summarize     | typed epistemic acts、phased session、shared workspace、tension tracking |
| 2 角色设计   | persona = 名字 + 性格 + "请你扮演 X" | 每个角色 = (允许 move × 必需证据 × 否决域 × 独有信息域) 四元组 |
| 3 Moderator  | 轮询调度 + 总结                      | 信息缺口探测、sycophancy 检测、阶段闸门、minority preservation、停止裁判 |
| 4 反玩具机制 | 无                                   | 至少 10 条硬要求全部实现（详见 §04）                         |
| 5 项目记忆   | 聊天记录 + 向量检索                  | 多账本结构 + provenance + propose-promote 工作流             |
| 6 讨论流程   | 单一线性流程                         | 七阶段模型 + 阶段间硬门控 + 自适应颗粒度                     |
| 7 输出结构   | Markdown 总结                        | 标准化 Decision Packet（含 minority report 与 reopen conditions） |

**这张表就是产品的可指责性源头**：随时可以拿任何一个真实会话和这张表对照，指出在哪一格上不达标。如果团队内部不愿意被这张表指责，就说明产品仍然在玩具区间。

### 0.3 为什么 2026.05 是合适的窗口

到这个时间点为止：

- 学术上，*From Debate to Deliberation* (2026.03) 第一次给出了 deliberation 的完整数学骨架——4 archetypes、14 typed epistemic acts、phased session、convergent flow 算法、structured decision packet。这一年里 Wynn et al. (*Talk Isn't Always Cheap*, 2509.05396)、Yao et al. (*Peacemaker or Troublemaker*, 2509.23055)、Choi et al. (2025.10) 等失效模式论文又把"什么会让多代理崩塌"扫得相当干净。**理论地基铸成了。**
- 工程上，Anthropic 的 multi-agent research blog (2025.06) 公开了第一份生产级 orchestrator-worker 经验；Cognition 的 *Don't Build Multi-Agents* (2025.06) 给出了反方边界。这两份合起来定义了"什么场景适合多代理 / 什么场景不适合"。
- 市场上，到 2026.05 为止没有任何一个开源或商业项目真正达到上面那张表的专业级标准。`askbudi/roundtable` 是 multi-CLI 调度器（不是 deliberation），`yorak/airoundtable` 是个简陋原型，`lijigang/ljg-roundtable` 是单场次哲学探讨 prompt skill（无项目记忆）。

**窗口期不会很长。** 这条赛道一旦被任何一家头部团队用对方法做出来，后入者很难追。

## 01 / 第一支柱：从 Debate 升级到 Deliberation

### 1.1 引入 typed epistemic acts（类型化认知动作）

目前你的设计里，角色发言是"自由文本"。这是玩具与专业的第一个分水岭。专业级 deliberation 要求每一句发言**显式标注它在做什么类型的认知动作**。这不是装饰，是后续 tension tracking、sycophancy detection、minority preservation 三件事的前提。

借鉴 DCI 的 14 类，按中文产品落地的可读性原则，建议精简为 **10 类**：

| 编码        | 名称     | 说明                                      | 谁可发起（参考）                 |
| ----------- | -------- | ----------------------------------------- | -------------------------------- |
| `CLAIM`     | 主张     | 提出一个立场或判断                        | 任何角色                         |
| `EVIDENCE`  | 证据     | 提供支持/反驳的事实、数据、来源           | 衡 主导，其他可补充              |
| `OBJECTION` | 异议     | 指出某主张的逻辑/事实缺陷                 | 砺 主导，其他可补充              |
| `REFINE`    | 精炼     | 在保留意图下重写一个主张以更精确          | Moderator + 提出者               |
| `CONCEDE`   | 让步     | 承认对方某点正确并修正自己                | 任何角色                         |
| `RESERVE`   | 保留     | 不同意但暂不阻断，登记为 residual tension | 任何角色                         |
| `ANALOGY`   | 类比     | 引入跨域类比作为启发                      | 漾 主导                          |
| `EMPATHY`   | 共情视角 | 从用户/受影响方角度提出考量               | 澈 主导                          |
| `PROBE`     | 探询     | 主动提问以挖掘未浮现的信息                | Moderator 主导                   |
| `META`      | 元       | 关于流程本身的意见（"我们是否跑偏了"）    | 任何角色，Moderator 决定是否打断 |

**实现层面**：每条发言在生成时必须输出结构化 JSON `{ "act": "OBJECTION", "target": "msg_id_xxx", "content": "...", "confidence": 0.65 }`。前端可以选择是否展示标签，但**数据层必须有**。这是后续所有机制的钩子。

### 1.2 Shared workspace 与 tension tracking

DCI 的核心创新不是分阶段，是 **shared workspace**：一个所有角色都能读写的结构化空间，记录的不是聊天记录，而是当前会话中"在讨论什么、还没解决什么、谁反对什么"。

最小可用 schema：

```
SessionWorkspace {
  topic: string
  candidate_options: Option[]      // 当前在评估的选项
  open_questions: Question[]       // 已浮现但未回答的问题
  tensions: Tension[]              // 未消解的分歧（每个挂着提出者、反对者、状态）
  evidence_pool: Evidence[]        // 已引入的证据，带 provenance
  pending_revisions: Revision[]    // 等待确认的主张修正
}

Tension {
  id: string
  description: string
  parties: { agree: RoleId[], disagree: RoleId[] }
  status: "open" | "resolved" | "deferred"
  resolution: string | null
}
```

**关键规则**：tensions 的 status 不允许由提出者单方面改成 resolved，必须经 Moderator 确认双方都接受当前的消解方案。这一条阻断了"主导角色把异议悄悄抹掉"的塌缩路径。

### 1.3 与单纯辩论的区分点

辩论的目标是说服与胜出；审议的目标是**整合分布式信息后做出可问责的决定**。后者在工程上的差别有四：

1. 发言必须类型化（§1.1）；
2. 分歧不被强制消解，而是被登记为 tension（§1.2）；
3. 修订必须基于新证据，不允许"因为对方更自信所以我改"（这是 Wu et al. 2025.11 的硬建议）；
4. 输出不是"谁赢了"，是 Decision Packet（§07）。

## 02 / 第二支柱：从 Persona 到 Epistemic Function

### 2.1 问题诊断：当前角色定义是装饰性的

把目前六个角色摊开看：

- 渡：主持人·哲学引导者
- 灼：远见·可能性主义者
- 砺：怀疑·风险拆解师
- 衡：数据·证据驱动者
- 澈：共情·用户视角守门人
- 漾：跨界·类比联想者

这是"性格清单"，不是"认知分工"。问题在于：**这些标签都可以由同一个 LLM 模拟出来，只换语气**。一旦模型推理松一点（比如 token 预算紧、上下文长、温度低），五个角色就会迅速向同一种"通用助手"塌缩。这是 MAD 失效模式研究里最普遍的现象（Wynn 2025；Choi 2025）。

要让角色真正不可替代，必须从 persona 升级到 **epistemic function**——把每个角色定义为一个 **四元组**：

```
Role = (
  allowed_moves:        EpistemicAct[],   // 这个角色被允许发起哪些 act
  required_evidence:    EvidenceType[],   // 它的 CLAIM 必须附带哪类证据
  veto_domain:          string[],         // 它在哪些类别的问题上有否决权
  unique_info_source:   InfoChannel[]     // 它独有的信息来源（参考资料、工具、检索域）
)
```

四元组里每一栏都是**可被外部检查的硬约束**，而不是"性格描述"。

### 2.2 六个角色的重定义（参考实现）

#### 渡（Moderator）

```
allowed_moves:        [PROBE, REFINE, META, 所有阶段闸门动作]
required_evidence:    无（Moderator 不发起 CLAIM）
veto_domain:          [阶段转移, 终止判定, sycophancy 干预]
unique_info_source:   [SessionWorkspace 完整读写权, 项目长期记忆完整读权]
```

渡**不参与论点本身**。它的发言永远不是"我认为 X"，而是"砺，你的 OBJECTION 没有附证据，请补充或撤回"、"我们现在累积了 3 个 unresolved tension，建议进入 Examine 阶段"、"灼和漾在上两轮都在重复同一论点，本轮请暂停"。详见 §03。

#### 灼（Visionary）

```
allowed_moves:        [CLAIM, ANALOGY, REFINE, RESERVE]
required_evidence:    [类比域案例 | 趋势数据 | 假设场景]（不允许零证据 CLAIM）
veto_domain:          [无]
unique_info_source:   [跨行业案例库, 趋势报告检索]
```

灼**不能发起 OBJECTION**——这一条很关键。远见者的价值在于扩展可能性，让它去否定别人会破坏分工。它表达不同意的方式只能是 RESERVE 或 REFINE。

#### 砺（Skeptic）

```
allowed_moves:        [OBJECTION, PROBE, RESERVE, CONCEDE]
required_evidence:    [反例 | 失败模式案例 | 逻辑反驳路径]
veto_domain:          [风险声明的最终封装]  // Decide 前砺有一次硬阻断权
unique_info_source:   [风险案例库, 失败复盘库, 红队 prompt 集]
```

砺**不发起 CLAIM**——它的工作是检验别人的 CLAIM。这一条把"怀疑论者变成另一个观点提供者"的塌缩堵死了。同时给砺一个 veto_domain：在 Decide 前，如果还有未解决的风险型 tension，砺有一次"硬阻断"权，强制 Moderator 回到 Examine 或要求用户介入。

#### 衡（Evidence Anchor）

```
allowed_moves:        [EVIDENCE, OBJECTION（仅当事实错误时）, REFINE, CONCEDE]
required_evidence:    [可引用来源 | 量化数据 | 一手观察]
veto_domain:          [事实性陈述]  // 衡可标记任何 CLAIM 为"未验证"
unique_info_source:   [外部检索, 数据库连接, 用户上传文档检索]
```

衡是**事实层的守门人**。任何 CLAIM 在被纳入 evidence_pool 之前，衡有标注权限：`unverified | partially-supported | well-supported | contradicted`。

#### 澈（Empathy Watch）

```
allowed_moves:        [EMPATHY, OBJECTION（仅当受影响方被忽视时）, RESERVE]
required_evidence:    [用户场景 | 受影响方画像 | 实际使用痕迹]
veto_domain:          [用户体验影响声明]
unique_info_source:   [用户访谈记录, 客服日志, 行为数据]
```

#### 漾（Cross-domain Bridge）

```
allowed_moves:        [ANALOGY, CLAIM（仅在引入新框架时）, REFINE]
required_evidence:    [源域案例 + 映射说明 + 失配警示]
veto_domain:          [无]
unique_info_source:   [跨学科案例库, 历史先例库]
```

漾被强制要求**每个类比都附带 failure-mode 警告**（"这个类比在 X 维度上不适用"）。没有失配警示的类比是不被接纳的——这堵死了"AI 编造好听的类比"这条最常见的玩具路径。

### 2.3 角色间 move 权限矩阵

```
              CLAIM  EVIDENCE  OBJECTION  REFINE  CONCEDE  RESERVE  ANALOGY  EMPATHY  PROBE  META
渡                                          ✓                                                ✓     ✓
灼              ✓                            ✓                ✓        ✓
砺                                ✓                  ✓        ✓                              ✓
衡              ✓*       ✓       ✓**       ✓        ✓
澈                                ✓***                       ✓                       ✓
漾              ✓****                       ✓                          ✓
```

\* 衡的 CLAIM 必须附数据 ** 衡的 OBJECTION 仅限事实错误 *** 澈的 OBJECTION 仅限受影响方被忽视 **** 漾的 CLAIM 仅在引入新框架时

这张矩阵直接转化为每个角色的 system prompt 约束。任何越权 move 由 Moderator 截获并要求重写或转换 act type。

### 2.4 异构模型的角色分配

既然你确认了走异构模型路线（Wu et al. 2025.11 强建议的"balanced heterogeneous teams"），下面是建议的初始配置：

| 角色 | 推荐模型类型                                         | 理由                                     |
| ---- | ---------------------------------------------------- | ---------------------------------------- |
| 渡   | 最强推理模型（如 Claude Opus 4.7 / GPT-5 类）        | Moderator 是整场会话的瓶颈，必须用最强的 |
| 砺   | 强推理 + 长上下文（Opus / Gemini 2.5 Pro）           | 反例检索和逻辑反驳需要细致推理           |
| 衡   | 长上下文 + 检索能力（Gemini 2.5 / 带 web 的 Claude） | 主要在做证据获取与验证                   |
| 灼   | 创造性强的中等模型（Sonnet / GPT 类）                | 发散性高，无需最强推理                   |
| 澈   | 中等模型                                             | 主要做立场转换，不吃推理                 |
| 漾   | 创造性强 + 知识广（Sonnet / Gemini Flash）           | 跨域召回                                 |

**关键的反塌缩点**：渡、砺、衡 三个角色尽量来自**不同模型家族**。Choi et al. (2025.10) 的实验显示，同源模型间 sycophancy 几乎不可避免；异源模型显著降低这种塌缩。

## 03 / 第三支柱：Moderator 是审议引擎而非主持人

### 3.1 问题诊断："哲学引导者"是危险的浪漫化

把渡叫"哲学引导者"是产品文案，不是工程定义。Moderator 在严肃 deliberation 里是**整个系统的中央处理器**，承担最重的工作。HiddenBench (Li et al. 2025, arXiv:2505.11556) 的结论非常硬：多代理 LLM 在 hidden profile 任务上只能拿 30.1%，而单代理给完整信息可以拿 80.7%。**瓶颈不在 aggregation 或 inference，而在 action selection——代理不会主动选择能最大化信息增益的沟通行为**。Moderator 就是来解决这个的。

### 3.2 Moderator 的七项硬职责

| #    | 职责                  | 触发条件                            | 输出动作                                               |
| ---- | --------------------- | ----------------------------------- | ------------------------------------------------------ |
| 1    | 信息缺口探测          | 每个阶段进入时 + 每 N 轮            | 生成 PROBE 列表，定向问角色"你独有的视角是什么"        |
| 2    | Sycophancy 检测       | 一个角色 CONCEDE 时                 | 检查 CONCEDE 是否附带新证据；无证据 CONCEDE 被退回     |
| 3    | 阶段闸门              | 每阶段终止条件满足时                | 决定是否进入下一阶段，必要时调用人类闸门（§06）        |
| 4    | Minority preservation | 任何 tension 即将被关闭时           | 强制登记 minority position 进入 Workspace              |
| 5    | 停止裁判              | 每轮结束                            | 评估三项指标：边际信息增量、tension 解决率、token 预算 |
| 6    | 越权拦截              | 角色发起不在 allowed_moves 内的 act | 截获、转换或退回                                       |
| 7    | 重复抑制              | 同一论点出现 >2 次                  | 标记该角色本轮静默，跳到其他角色                       |

每项职责都对应一个 Moderator system prompt 中的显式段落与一个可检查的代码侧 hook。这不是"理念"，是 7 个具体的工程接口。

### 3.3 Moderator 的元 prompt 骨架（参考实现）

```
你是会话主持人。你不参与论点。

每轮你只做以下三类事：
1. 安排下一个发言者（基于 §06 的阶段规则 + 当前 Workspace 状态）
2. 验证上一条发言是否符合该角色的 allowed_moves 与证据要求
3. 维护 Workspace（特别是 tensions 与 evidence_pool）

每 3 轮，你必须执行一次主动信息探测：
- 扫描 candidate_options × roles 矩阵
- 找出 unique_info_source 与 candidate_options 有交集但该角色尚未贡献的格子
- 生成针对性 PROBE，强制对应角色回应

进入下一阶段前，你必须输出一份"阶段总结"：
- 本阶段产出的新 EVIDENCE 数（要求 ≥ X）
- 本阶段消解的 tension 数 / 新增 tension 数
- 本阶段是否触发任何 sycophancy 警报
- 推荐进入的下一阶段 + 备选方案
- 是否需要人类介入

如果你检测到下列任一情况，立即调用人类闸门：
- 连续 2 轮无新 evidence 且 tension 数不减
- 关键 candidate option 没有任何 OBJECTION 但接近 Decide
- 任何角色三轮内 CONCEDE 比例 > 60% 且 CONCEDE 都无新证据
- token 预算消耗 > 80%
```

### 3.4 Moderator 不该做什么

同样关键的是显式列出**禁区**：

- 不发起 CLAIM、EVIDENCE、OBJECTION、ANALOGY、EMPATHY
- 不评价某个论点的"正确性"——这是 Decide 阶段用户的工作
- 不总结别人观点然后追问"对吗"——这制造 sycophancy
- 不在中间阶段输出"最终建议"

把 Moderator 严格限制在流程层，是阻止它"占用论点空间"的关键。这条很多产品都做错了——主持人最后变成了第七个观点提供者。

## 04 / 第四支柱：反玩具机制（10 条硬要求）

下面 10 条每一条都来自一篇已发表的失效模式论文。这不是"建议"，是底线。没有这 10 条，产品会在任何稍微严肃的场景下原形毕露。

**M1 — 关键阶段匿名化发言归属** *来源：Choi et al., 2025.10* 实验显示**匿名化 response source 几乎完全消除 identity-driven sycophancy**。具体做法：在 Examine 阶段，角色发言对其他角色显示为"参与者 A / B / C"而非"灼/衡"，等阶段结束才揭示。前端 UI 上同样匿名，避免用户的"信任偏见"也参与塌缩。

**M2 — 置信度默认隐藏** *来源：Wu et al., 2025.11* 每条发言内部带 confidence，但**默认不展示给其他角色**，仅 Moderator 可见。理由：置信度可见会触发"confidence cascade"——其他角色锚定到高置信度发言，停止独立思考。仅在 Decide 阶段统一披露。

**M3 — 强制显式异议** *来源：Wu et al., 2025.11* 每轮结束时，每个未发言的角色被 Moderator 显式询问："你对本轮的主要 CLAIM agree / disagree / reserve？无论选什么都必须给一个理由"。沉默允许（§2.2），但**伪沉默**（"我同意大家说的"）被退回。

**M4 — Devil's advocate 注入** *来源：Liang et al., 2023; Wynn et al., 2025* 当连续 3 轮没有出现任何 OBJECTION 时，Moderator 强制注入一个"反方角色"——通常是临时让砺以 PROBE 模式提出"我们是不是漏了什么重要反驳"。这是 echo chamber 最有效的破解器。

**M5 — Premature consensus 阻断** *来源：Yao et al., 2025.09* 进入 Decide 阶段前，必须经过至少一个 minimum tension exploration window：在该窗口内 Moderator 主动列出"已知但未充分讨论的反对意见"，强制每个角色回应。这堵死"AI 提前收敛"——deliberation 系统最常见的退化路径。

**M6 — Hidden information probing** *来源：Li et al., 2025; "Bringing Everyone to the Table", 2024* 每个角色的 `unique_info_source` 在会话开始时被注册。Moderator 必须保证每个 source 至少被 PROBE 一次。HiddenBench 的核心教训：**没有主动探测，分布式信息无法浮现**。

**M7 — 修订必须基于新证据** *来源：Wu et al., 2025.11* 角色 CONCEDE 或 REFINE 自己的早期 CLAIM 时，必须显式引用触发修订的 EVIDENCE 或 OBJECTION 的 message_id。无引用的修订被退回。这阻止"因为对方更自信所以我改"的盲目顺从。

**M8 — Token / 轮次预算 + sparsification** *来源：ICLR 2025 MAD blog; Li et al., 2024* 每场会话开始时设定 token 预算和最大轮次。轮次过半仍未进入 Examine 时触发 Moderator 自检。同时实现 sparsification：在大型会话中（>4 角色），允许 Moderator 只向相关角色广播某条发言而不是全员广播。Li et al. (2024) 的数据：sparsification 可节省高达 94.5% token 且精度损失 <2%。

**M9 — Minority report 强制保留** *来源：DCI 2026; Wynn et al., 2025* 任何 tension 在 Decide 时未被消解，必须以 minority report 的形式进入 Decision Packet。Decision Packet 中**没有 minority report 段落是不允许导出的**——前端校验。这是 deliberation 最重要的工程契约。

**M10 — Provenance 强制** *来源：Mem0 / Atlan 2026; Anthropic Research 2025* 任何 EVIDENCE 必须带来源（URL / 文档 ID / 工具调用 ID / 用户上传文件位置）。无来源的事实性陈述自动被衡标注为 `unverified`，不进入 evidence_pool。Decision Packet 中所有引用的事实必须可点回到来源。**这是用户能否信任产品的根**。

## 05 / 第五支柱：项目记忆架构

### 5.1 三层 scope

你已经选择"项目为单位"的方向，这正好对齐 Mem0 三层 scope（user / agent / run）的产业共识。具体到 AI Roundtable Room：

```
LAYER 1: USER LAYER         // 跨项目的用户偏好
  - 沟通风格偏好
  - 默认场景配置
  - 角色定制偏好（例如该用户希望砺更激进）

LAYER 2: PROJECT LAYER      // 项目级长期记忆（核心）
  - 见 §5.2 六账本

LAYER 3: SESSION LAYER      // 单场会话工作区
  - 见 §1.2 SessionWorkspace
```

Session 结束时由用户复核后**部分回写**到 Project Layer。这是 propose-promote 工作流的物理边界。

### 5.2 项目记忆的六个账本

这是整个产品最容易做错的地方。绝大多数玩具产品把项目记忆做成了"过去对话的向量索引"。这等于没做。专业级最低标准是**六张结构化账本**：

```
ProjectMemory {
  decisions_ledger:    Decision[]       // 已做出的决定（带场次溯源）
  risk_register:       Risk[]           // 已识别的风险（status: open/mitigated/accepted）
  open_questions:      Question[]       // 尚未回答的关键问题
  action_items:        Action[]         // 待执行项（带 owner、deadline、status）
  evidence_pool:       Evidence[]       // 项目级证据库（去重后的）
  residual_tensions:   Tension[]        // 跨场次保留的未解张力
  reopen_conditions:   Condition[]      // 触发哪些条件需重开某决定
}

Decision {
  id: string
  question: string                       // 决定回答了什么问题
  selected_option: string
  rationale: string
  alternatives_considered: Option[]
  minority_report: string | null
  session_id: string                     // 在哪场会话做出
  date: timestamp
  reopen_conditions: ConditionId[]
  related_evidence: EvidenceId[]
  status: "active" | "revised" | "obsolete"
}

Tension {
  id: string
  description: string
  parties: { positions: Map<RoleId, string> }
  attempts_to_resolve: SessionId[]       // 哪些场次尝试过解决
  current_status: string
  related_decisions: DecisionId[]
}
```

每张账本的关键设计：

- **decisions_ledger**：每个 Decision 都有 `reopen_conditions`。这是 DCI 的核心创新——决定不是"封死的"，而是"在 X 条件下需要重审"。例如"如果竞品在 Q2 前推出 Y 功能，重审上线优先级"。
- **risk_register**：每条 Risk 都有 status，可以跨场次更新。第二场会话开始时，渡会主动 surface "上次还有 3 个 open 风险，我们今天要不要处理"。
- **residual_tensions**：跨场次保留的分歧。这是普通产品最容易丢失的资产。
- **reopen_conditions** 单独抽出来，因为它跨多个 Decision 共享。

### 5.3 Propose-Promote 工作流

借鉴 Mem0 + Atlan 2026 的硬经验："*agent's job is to propose, not to publish*"。

```
Session 结束流程：
1. Moderator 生成"建议入库的项目记忆变更"列表
2. 用户在 Memory Review UI 中逐项 approve / edit / reject
3. 用户批准的变更才进入 Project Layer
4. 被拒绝的变更进入 rejected/ 归档（保留用于后续模型微调）
```

**绝对不能**让 AI 直接写项目长期记忆。这是产品级事故的源头——一旦 AI 自由写记忆，第三场会话的 AI 看到的就是第二场会话 AI 编造的"事实"，错误会指数级放大。

### 5.4 个人 vs 团队场景的差异

你确认了"两者兼有"，所以项目记忆必须支持：

- **个人模式**：单一 owner，所有变更直接通知该用户
- **团队模式**：项目有多个 collaborator；记忆变更可以由任一 collaborator 在 Memory Review 中 approve，但敏感字段（决定、风险）的变更需要 ≥ 2 人 approve；所有变更带 actor_id 和 timestamp，可审计

团队模式默认在 V0.5 引入，V0.1 仅个人模式即可（见 §10）。

### 5.5 与 Session Layer 的写入契约

Session 结束时，下列字段**自动建议**写入 Project Layer（仍需用户复核）：

- 任何 `Decision`（必入）
- 状态变为 `mitigated` 或 `accepted` 的 `Risk`
- 新增的 `Risk`
- 任何 `Action`
- 标记为 `well-supported` 的 `Evidence`（去重）
- 未消解的 `Tension`（进入 residual_tensions）

下列字段**不自动写入**：

- 会话内 chat history（仅在 session log 中保留，可检索但不视为"事实"）
- 被衡标注为 `unverified` 的 CLAIM
- Moderator 的流程性发言

## 06 / 第六支柱：阶段化流程

### 6.1 七阶段通用模型

```
Frame  →  Diverge  →  Surface  →  Examine  →  Converge  →  Decide  →  Document
```

每阶段的规格：

| 阶段         | 主任务                 | 主导角色                    | 允许 act                    | 终止条件                                           |
| ------------ | ---------------------- | --------------------------- | --------------------------- | -------------------------------------------------- |
| **Frame**    | 明确议题与边界         | 渡 + 用户                   | PROBE, META                 | 议题 + 决策类型 + 利益相关方 三件齐全              |
| **Diverge**  | 扩展候选选项           | 灼、漾 主导；砺/衡 静默     | CLAIM, ANALOGY, REFINE      | candidate_options ≥ N，且新增速率衰减              |
| **Surface**  | 浮现独有信息           | 全员 + Moderator 强制 PROBE | EVIDENCE, EMPATHY, PROBE    | 每个 unique_info_source 至少被触发一次             |
| **Examine**  | 检验 + 制造张力        | 砺、衡 主导；灼/漾 受限     | OBJECTION, EVIDENCE, REFINE | 每个 candidate option ≥ 1 OBJECTION + 1 EVIDENCE   |
| **Converge** | 缩小到可决定的少数选项 | 渡 + 全员                   | REFINE, CONCEDE, RESERVE    | candidate_options 缩减到 ≤ 3，且每个有完整 pro/con |
| **Decide**   | 选择 + 封装            | 渡 + 用户（人类闸门）       | ——                          | 用户在 Decision Packet 上确认                      |
| **Document** | 回写项目记忆           | 渡 + 用户                   | ——                          | Memory Review 完成                                 |

### 6.2 阶段间硬门控

四个不可跳过的门控：

- **Frame → Diverge**：议题必须经过用户确认。无确认不进入下一阶段。
- **Diverge → Surface**：候选选项数 ≥ N。这阻止"AI 太快收敛"。
- **Examine → Converge**：每个 candidate option 必须有至少 1 个 OBJECTION 与 1 个 EVIDENCE，否则不允许进入收敛。这是 M5 的具体落地。
- **Converge → Decide**：M9 强制——所有未消解 tension 必须显式登记为 minority position。

### 6.3 自适应颗粒度

你确认了"会话长度取决于问题"。设置三档预设：

| 档位     | 适用           | 总轮次  | 七阶段时间分布    |
| -------- | -------------- | ------- | ----------------- |
| **轻量** | 5–10 分钟探讨  | ≤ 15 轮 | 1/3/2/3/2/2/2     |
| **标准** | 20–30 分钟评审 | ≤ 40 轮 | 2/6/5/10/6/4/3    |
| **深度** | 60+ 分钟审议   | ≤ 80 轮 | 3/12/10/20/15/8/5 |

档位由 Frame 阶段结束时由 Moderator 推荐 + 用户确认。运行中允许"升档"（从轻量升标准），不允许"降档"（这会触发已有 tension 的丢弃）。

## 07 / 第七支柱：Decision Packet 输出标准

### 7.1 字段定义

```yaml
DecisionPacket:
  packet_id: string
  session_id: string
  project_id: string
  topic: string
  date: ISO-8601

  selected_option:
    id: string
    description: string
    rationale: string                    # 选这个的理由（基于哪些 evidence）
    confidence: float                    # 此时披露 §M2

  alternatives_considered:
    - id: string
      description: string
      why_not_selected: string

  residual_objections:                   # M9 强制
    - objection: string
      raised_by: RoleId
      addressed_by: string | null        # 如何应对，可为 null
      acceptable_to_decider: bool

  minority_report:                       # M9 强制
    position: string | null              # 持少数意见的角色 + 立场
    rationale: string | null
    conditions_to_reconsider: string[]

  reopen_conditions:                     # 触发重审的条件
    - condition: string
      check_mechanism: string            # 如何/何时检查

  evidence_used:                         # M10 强制
    - id: string
      source: string                     # 必须可追溯
      verification_status: enum

  action_items:
    - action: string
      owner: string
      deadline: ISO-8601 | null

  open_questions:                        # 未回答但与此决定相关
    - question: string
      blocks: bool                       # 是否阻碍执行

  participants:
    moderator: { role: "渡", model: string }
    panelists:
      - { role: RoleId, model: string }
```

### 7.2 渲染规则

- 产品内展示：可折叠，但 `minority_report` 与 `residual_objections` **默认展开**。隐藏少数意见是玩具产品的标志。
- 导出格式：Markdown / PDF / JSON 三选一。JSON 用于回写项目记忆与机器后处理。
- 任何字段缺失（特别是 minority_report、reopen_conditions）的 packet 不允许 finalize——前端阻断。

### 7.3 与项目记忆的回写契约

Decision Packet finalize 时，自动生成一份"建议入库变更清单"提交 Memory Review（§5.3），不允许跳过。

## 08 / 场景差异化矩阵

参数化人数远远不够。四类场景在**机制层**有本质差异：

| 维度                       | 头脑风暴                       | 产品深度评审                | 新领域研究                    | 风险/合规审查                          |
| -------------------------- | ------------------------------ | --------------------------- | ----------------------------- | -------------------------------------- |
| **核心目标**               | 候选空间最大化                 | 选项收敛 + 风险标识         | 知识获取 + 假设建立           | 风险穷尽 + 合规校验                    |
| **阶段权重**               | Diverge 主导，几乎跳过 Examine | Surface + Examine 主导      | Surface 主导，多次 Frame 迭代 | Examine 主导，强制 red-team            |
| **必参角色**               | 灼、漾、澈                     | 全员                        | 衡、灼、漾                    | 全员 + 临时增设"对手方"                |
| **静默角色（默认）**       | 砺、衡（前半程）               | 无                          | 砺（前半程）                  | 无                                     |
| **Moderator 优先目标**     | 抑制过早批判                   | 平衡 divergence/convergence | 主动 PROBE 知识缺口           | 穷举 OBJECTION 路径                    |
| **匿名化（M1）**           | 关                             | Examine 阶段开              | 关                            | 全程开                                 |
| **devil's advocate（M4）** | Examine 时触发                 | 标准触发                    | 标准触发                      | 强制每轮触发                           |
| **证据要求**               | 弱（允许 ANALOGY 替代）        | 标准                        | 高（衡的标注权严格）          | 极高（无来源即拒绝）                   |
| **停止准则**               | 候选选项新增速率衰减           | 三选项 pro/con 齐全         | open_questions 列表稳定       | 所有 candidate 至少 N 个独立 OBJECTION |
| **必出 minority report**   | 不强制                         | 强制                        | 强制                          | 强制 + 必须给 "如果此风险成真" 应急    |

**关键警告**：风险/合规审查模式必须把"对手方"作为临时第七角色注入。砺再尖锐也是"自己人提风险"，不能替代真正的红队思维。

## 09 / 当前设计诊断（可指责清单）

按上面七维度对照当前给出的设计，按"必须修 / 应该修 / 可以延后"分档：

### 必须修（V0.1 前完成）

1. **角色定义从 persona 升级到 epistemic function 四元组**（§02）。当前的"远见者/怀疑者/数据派/共情者/跨界者"是装饰，需要重写为 allowed_moves × required_evidence × veto_domain × unique_info_source。
2. **Moderator 重新定义**（§03）。"哲学引导者"必须替换为承担七项硬职责的审议引擎。
3. **引入 typed epistemic acts**（§01）。这是后续所有机制的钩子，不引入则所有反玩具机制都无法实现。
4. **Decision Packet 输出标准**（§07）。当前没有显式输出契约，这是用户复购的关键。
5. **项目记忆六账本 schema**（§5.2）。当前"项目为单位"只有意图，没有形状。
6. **Propose-Promote 工作流**（§5.3）。直接让 AI 写长期记忆会爆炸。
7. **七阶段流程 + 硬门控**（§06）。

### 应该修（V0.5 前完成）

1. **10 条反玩具机制全部上线**（§04）。V0.1 至少落 M1、M5、M6、M9、M10 五条核心；其余补全到 V0.5。
2. **场景差异化矩阵**（§08）。当前的"人数差异化"是错的，需要按机制差异化重做。
3. **异构模型角色分配**（§2.4）。
4. **团队场景支持**（§5.4）。

### 可以延后（V1.0 后）

1. 跨项目模式发现（"你过去三个项目里都有 X 类型的风险被忽略"）
2. 角色定制（用户自定义新角色）
3. 对手方临时角色注入的 UI（§08 风险审查模式）

## 10 / 阶段性演进路径

### V0.1 — 最小可用专业版（目标：4–6 周）

**目标**：在标准档（20–30 分钟）下，跑通一场"产品深度评审"完整流程，输出合格 Decision Packet。

**必含**：

- 七阶段流程 + 三个硬门控
- 六角色 epistemic function 重定义（基础版，矩阵简化）

- 项目记忆 decisions_ledger + risk_register 两张账本

**不含**：异构模型、匿名化、团队模式、深度档、风险审查模式。

## 附录 B / 与 ljg-roundtable 的协作边界

ljg-roundtable 是 Claude Code Skill，特征：

- 单场次、纯 prompt（无运行时状态）
- 角色 = 真实历史/当代人物（如丹尼特、萨特、斯宾诺莎、李白）
- 求真导向、挖深不铺广、每轮一条最深的裂缝
- ASCII 思考框架图（拓扑/矩阵/光谱/树形）
- org-mode 输出
- 适合"哲学探讨型 ad-hoc 议题"，**无项目记忆**

AI Roundtable Room 与之的关系：

**互补而非替代**。

- ljg-roundtable 优势：人物真实性带来的认知差异度更高、ASCII 图的"见骨"呈现、求真姿态
- AI Roundtable Room 优势：项目锚定、跨场次记忆、Decision Packet 可问责输出、四类场景差异化

**集成建议**（V1.0 阶段）：

1. **将 ljg-roundtable 作为"探索性议题"模式接入**。当用户在某项目下提出一个仍处于"问题定义不清"的议题时，可临时切换到 ljg-roundtable 风格的真实人物圆桌做一场探索，但最终的 Decision Packet 仍按 AI Roundtable Room 标准产出。
2. **借鉴 ljg-roundtable 的 ASCII 张力图作为 Examine 阶段的呈现方式**。在 SessionWorkspace 的 tensions 列表渲染时，可同时输出一张张力图。
3. **不要混用角色体系**。ljg-roundtable 用真实人物，AI Roundtable Room 用 epistemic function 抽象角色。两者各有所长，混用会破坏 epistemic function 的可指责性。
4. **共享 prompt 工艺**。ljg-roundtable 在"求真 > 和谐"这条姿态上写得很好，可以在 Moderator 的反 sycophancy 段落里借鉴具体措辞。

## 附录 C / 评估指标（如何客观判断"我们是不是玩具"）

每个版本上线前跑这套指标，分数公开。

### C.1 流程合规度

- **act 类型覆盖率**：每场会话出现的 epistemic act 类型 / 10 类。目标 ≥ 7
- **角色 silence 出现率**：至少有一个角色在至少一个 phase 中保持沉默的会话占比。目标 ≥ 40%（沉默是健康信号）
- **Moderator 越权拦截次数**：每场平均拦截数。目标 1–5 之间（过低=没在工作；过高=角色 prompt 出问题）

### C.2 失效模式抑制

- **Sycophancy 警报率**：每场触发 M2/M3/M7 警报的次数。**追踪趋势是否下降**。
- **Devil's advocate 触发率**：M4 在 Examine 阶段被触发的会话占比。目标 30–60%。
- **Premature consensus 阻断率**：M5 阻断进入 Decide 的次数。下降说明角色辩证能力上升。

### C.3 输出质量

- **Decision Packet 完整率**：所有字段填满且 minority_report 非空的 packet 占比。目标 100%。
- **Provenance 完整率**：Decision 中所有 evidence_used 可追溯到来源。目标 100%。
- **Reopen condition 命中率**：上线后被实际触发并启动重审的 reopen_condition 数。

### C.4 用户侧（人类闸门质量）

- **Memory Review 修改率**：用户在 propose-promote 中编辑的比例。**目标在中间**——过低说明用户没在认真审，过高说明 AI 提案质量差。
- **Decision Packet 二次使用率**：用户在后续场次中引用前场 Decision 的频率。这是产品价值最硬的指标。
- **回访率**：项目第二场、第三场、第 N 场会话的留存。**这是判断"是不是玩具"的最终裁判**。

### C.5 HiddenBench 风格内部测试

每个版本运行一组 hidden profile 任务（参考 arXiv:2505.11556 公开 65 题），跑 30 次平均。专业级最低标准：**多场次（带项目记忆）的准确率 > 65%**（无记忆 LLM 多代理基线约 30%；单代理给完整信息约 80%）。这条指标过不了，产品没必要发版。
