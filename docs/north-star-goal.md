# AI Roundtable Room — North Star Goal

## 一句话目标

AI Roundtable Room 的目标不是模拟多人圆桌会议，而是构建一个 AI-native 的认知增强系统：围绕用户当前真正关心的问题，动态组织多个高质量判断函数进行受控碰撞、冲突收束和用户介入，帮助用户获得更好的问题、更清晰的分歧、更可靠的判断，以及更可执行的下一步行动。

## 核心定义

圆桌会议不是多个 Agent 轮流聊天。

圆桌会议是把一个问题放进多个高质量判断函数之间碰撞，让不同判断标准相互压迫、反驳、修正和收束，最终产出被挑战过的观点、核心分歧、重构后的问题和下一步选择。

在这个项目里，“圆桌”只是用户容易理解的界面隐喻，不是系统必须复制的组织形式。系统不应该追求会议拟真度，而应该追求认知增益。

## 我们真正要提升什么

用户来这里不是为了看 AI 表演开会，而是为了让自己的思考变强。

一个有效圆桌应该提升：

1. **问题质量**：用户原来的问题是否被重新定义得更准确？
2. **注意力质量**：用户是否看到原本忽略但更关键的切入点？
3. **判断质量**：结论是否经过反方、证据、风险、用户价值和落地约束的挑战？
4. **分歧质量**：系统是否保留了真正影响决策的冲突，而不是抹平成虚假共识？
5. **行动质量**：用户是否得到可继续推进、验证或决策的下一步？

## 圆桌的本质机制

高质量圆桌由三个机制组成。

### 1. 多判断函数

系统可以有角色，但角色不是重点。重点是每个参与视角必须代表一种清晰、必要、不可互相替代的判断函数。

例如：

- 第一性原理重构：这个问题的真实结构是什么？
- 用户心智判断：真实用户会如何理解、误解或使用？
- 商业与战略判断：这件事是否值得做，优先级在哪里？
- 风险与反方攻击：什么地方可能错、假、贵、慢或危险？
- 证据与置信度校准：哪些结论有证据，哪些只是猜测？
- 执行落地判断：下一步能做什么实验、任务或决策？

如果两个角色只是语气不同、观点重复，它们不应该同时存在。

### 2. 受控碰撞

圆桌不是并列罗列观点，而是让观点发生有约束的碰撞。

受控碰撞意味着：

- 每个关键判断都要被至少一个不同视角挑战；
- 分歧必须被命名，而不是藏在聊天里；
- 冲突要区分事实冲突、价值冲突、风险偏好冲突和方案冲突；
- 无法解决的分歧要保留，并说明需要什么证据或用户选择；
- 系统要避免礼貌性赞同、快速和稀泥和虚假共识。

### 3. 收束与用户介入

圆桌不应该无限自动讨论。系统必须在关键节点停下来，把选择权交还给用户。

每轮圆桌都应该收束为：

- 当前最重要的判断；
- 仍未解决的分歧；
- 被推翻或需要验证的假设；
- 值得继续深挖的新问题；
- 用户可以选择的下一步路径。

用户不是观众，而是圆桌的最终决策者和注意力选择者。

## 什么不是我们的目标

本项目不追求：

- 为了热闹而增加角色数量；
- 为了像会议而轮流发言；
- 为了拟人而设计复杂人格；
- 为了显得专业而拉长对话轮数；
- 为了输出顺滑总结而抹平分歧；
- 为了“多 Agent”标签而强行拆分模型调用；
- 把圆桌做成普通聊天应用的换皮版本。

多 Agent 可以是实现手段，但不是产品目标。

如果单模型能更低成本地产生高质量判断函数、冲突和收束，就不应该为了形式感强行多 Agent。反过来，如果真实多 Agent 能带来独立性、并行性、模型差异和更强反方压力，就应该让多 Agent 服务于判断质量，而不是服务于表演。

## 高质量圆桌的最低标准

一场圆桌只有满足以下条件，才值得被称为有效圆桌。

### 1. 有明确准入理由

系统必须能解释为什么这个问题需要圆桌，而不是直接单模型回答。

适合圆桌的问题通常具有：

- 高价值决策；
- 多目标冲突；
- 不确定性；
- 风险或代价；
- 需要保留分歧；
- 需要项目级记忆和复盘。

简单事实查询、低风险文本润色、明确单步任务，不应该默认进入圆桌。

### 2. 有不可替代的视角差异

每个参与视角都必须贡献不同判断压力。没有认知差异的角色应该合并或删除。

### 3. 有真实冲突

系统必须找出影响判断的核心分歧。没有分歧不代表成功，可能代表系统没有充分挑战问题。

### 4. 有证据和置信度意识

圆桌不能把猜测包装成结论。每个关键判断都应该区分：

- 已知事实；
- 合理推断；
- 待验证假设；
- 主观价值判断。

### 5. 有结构化产物

最终产物不应该只是聊天记录，而应该包含：

- 重构后的问题；
- 关键判断；
- 核心分歧；
- 少数意见；
- 风险与假设；
- 行动项或实验；
- 下一轮值得追问的方向。

### 6. 有用户介入节点

系统必须在合适时机让用户选择继续深挖、收束决策、补充信息或切换方向。

### 7. 有可复盘性

圆桌结果必须能被之后的项目记忆、决策日志、风险清单和行动追踪复用，而不是沉没在一次性聊天里。

## Toy 与 Serious Product 的分界线

Toy 圆桌卖的是多角色表演。

Serious 圆桌卖的是结构化判断质量。

| 维度 | Toy 圆桌 | Serious 圆桌 |
| --- | --- | --- |
| 核心目标 | 看起来像多人聊天 | 提升用户判断质量 |
| 角色价值 | 人设、语气、戏剧感 | 不同判断函数和认知压力 |
| 讨论方式 | 轮流发言 | 受控碰撞、挑战、修正、收束 |
| 分歧处理 | 快速达成共识 | 显式保留冲突和少数意见 |
| 用户位置 | 旁观者 | 决策者和注意力选择者 |
| 输出形态 | 聊天记录和总结 | Decision Packet / 问题地图 / 下一步路径 |
| 成本意识 | 越长越像专业 | 每一轮都必须产生认知增益 |

## 产品设计原则

### 原则 1：先判断是否需要圆桌

不是所有问题都应该开圆桌。系统应先判断任务类型、价值、风险和不确定性，再决定走单模型、轻量多视角还是完整圆桌。

### 原则 2：角色服从判断函数

角色可以保留，但每个角色都必须有职责合约。角色不能只是名字、性格和语气。

### 原则 3：对话服从结构

聊天流不是最终产物。界面应该突出问题、分歧、证据、风险、行动和用户可选路径。

### 原则 4：冲突比和谐更重要

好的圆桌不急于达成共识。它应该保护那些影响决策的真实分歧。

### 原则 5：收束比延长更重要

专业感不来自长篇讨论，而来自适时停止、压缩、命名和给出选择。

### 原则 6：用户注意力是核心资源

系统每次输出都应该回答：这会让用户接下来更关注什么？如果不能改变用户的注意力或行动，就应该删减。

### 原则 7：记忆服务于下一次判断

项目记忆不是保存聊天历史，而是保存下次判断真正需要的结论、分歧、假设、风险和行动状态。

## 项目重塑方向

后续所有功能、UI、Prompt、数据结构和文档都应围绕这个 North Star 重塑。

优先方向：

1. 从“角色聊天流”转向“判断函数碰撞流”；
2. 从“圆桌记录”转向“结构化 Decision Packet”；
3. 从“AI 自动讨论”转向“关键节点用户介入”；
4. 从“多角色人设”转向“职责合约与认知压力”；
5. 从“历史对话保存”转向“项目级判断记忆”；
6. 从“输出一段总结”转向“生成下一步可选路径”。

## 判断任何改动是否正确的检查问题

每次新增或修改功能前，先问：

1. 它是否提升了用户的问题质量、判断质量或行动质量？
2. 它是否减少了无意义角色表演？
3. 它是否让关键分歧更清楚，而不是更模糊？
4. 它是否让用户更容易介入和选择方向？
5. 它是否能沉淀为项目级记忆或可复盘产物？
6. 它是否证明了圆桌比单模型回答更值得？

如果答案是否定的，就不应该做。

## 最终 North Star

AI Roundtable Room 要成为一个严肃的 AI-native 思考工作台。

它不复制人类会议的表面形式，而是提取圆桌真正有价值的部分：多视角压力、公开分歧、反复修正、共同收束和可复盘决策。

它的成功标准不是“AI 们聊得像不像人”，而是：

> 用户是否因为这场圆桌，看见了更关键的问题，理解了更真实的分歧，形成了更可靠的判断，并知道下一步该怎么推进。
