# AI Roundtable Room 改进建议书（v2）

**日期**：2026-05-17  
**版本**：v2（基于项目记忆定位调整）  
**核心定位**：本产品应定位为「**项目级 AI 结构化审议工作台**」，而非长期角色演化的圆桌空间。

---

## 1. 产品定位重新明确

### 1.1 正确的心智模型

**AI Roundtable Room 是一个为具体项目服务的结构化 AI 审议工具。**

- 一个项目可以反复开启多次圆桌会议（不同阶段、不同深度、不同议题）。
- 系统需要记住**这个项目**的历史讨论、关键决策、遗留风险和行动项。
- 每一次新圆桌，都应该能自动带入该项目的上下文。
- 角色是服务于项目的工具，而不是需要长期养成的人格。

**一句话定位**：
> 为每一个具体项目，提供一个可反复使用、带有项目记忆的、高质量 AI 结构化审议平台。

### 1.2 为什么这个定位更合理？

- 符合真实用户需求（复杂项目需要多次审议）。
- 避免了长期角色记忆带来的上下文污染、性格漂移和管理复杂度。
- 与 ljg-roundtable 的结构化辩证能力高度互补。
- 更容易做出差异化。

---

## 2. 当前主要问题（按优先级）

| 优先级 | 问题 | 说明 |
|--------|------|------|
| P0 | 缺少「项目」这个第一级概念 | 目前会议是孤立的，没有项目归属和项目上下文 |
| P0 | 项目记忆能力几乎为零 | 新开圆桌无法自动带入历史结论、风险和行动 |
| P1 | Moderator 缺少项目视角 | 只能管好单场会议，无法感知项目整体状态 |
| P1 | 输出结构化程度不够项目化 | 纪要可用，但缺少项目级聚合视图和追踪能力 |
| P2 | 角色系统偏静态 | 角色缺少项目内的立场和知识积累 |
| P2 | 观看与参与体验一般 | 长会议用户容易跟丢脉络 |

---

## 3. 改进核心原则

1. **项目是第一实体**：所有功能设计都要围绕「项目」展开。
2. **记忆以项目为单位**：记忆必须可控、可结构化、可被新圆桌引用。
3. **Moderator 必须有项目意识**：同时服务单场质量和项目整体审议健康度。
4. **输出要能反哺记忆**：圆桌成果要结构化沉淀回项目。
5. **保持克制**：优先把项目记忆和 Moderator 两条线打穿。

---

## 4. 分阶段改进计划

### Phase 1：建立项目记忆基础（最优先）

- 增加「项目」概念（创建、切换、归档项目）。
- 新开圆桌时可选择项目，并自动注入该项目的历史关键信息（上次结论、未闭合风险、待办行动）。
- 会议结束后，将结构化纪要中的关键内容反向写入项目记忆。
- 提供项目级视图（历史圆桌 + 风险清单 + 行动追踪）。

### Phase 2：升级 Moderator（项目级 Moderator）

- 重构 Moderator 提示框架，加入项目上下文。
- 让 Moderator 能够引用项目历史、追踪行动项闭环情况。
- 增加项目级总结能力。

### Phase 3：输出与记忆深度融合

- 优化会议纪要结构，便于被项目记忆系统提取。
- 增加「项目级阶段性总结」功能。
- 支持上传项目文档作为长期上下文。

### Phase 4：体验与工程打磨

- 提升观看体验（时间线、高亮、折叠）。
- 成本可视化与长会议上下文管理。
- 支持不同审议模板（战略型、风险型、技术评审型等）。

---

## 5. 与 yzc-skills / ljg-roundtable 的结合

- 建议将 ljg-roundtable 的结构化辩证思维框架，迁移到本项目的 **Moderator 和角色提示** 中。
- 把 yzc-drive 的执行纪律用于长项目的行动追踪和风险闭环提醒。
- 未来可考虑将部分高质量 roundtable 逻辑沉淀到 yzc-skills 中统一维护。

---

## 6. 风险与取舍

- 暂不做跨项目的长期角色记忆。
- 项目记忆以**结构化提炼**为主，而不是原始对话堆砌。
- 先做最小可行版本验证，再决定是否继续加功能。

---

## 7. 下一步建议

1. 确认是否完全同意「项目记忆」为主的定位。
2. 确定 Phase 1 的最小范围。
3. 设计项目记忆的数据结构（建议 JSON + Markdown 双存储）。
4. 开始重构 Moderator 提示框架（可结合 ljg-roundtable 一起设计）。
